Threadlight / Self-improving

The pipeline that learns from every run

Every threadlight CI run — green or red — carries learnings, and re-reading raw logs by hand is slow and noisy. So a cold-path called learn harvests one GitHub Actions run, no matched-pair baseline, and turns it into a grounded list of ranked fixes — each one mapped back to the evidence that justifies it. Point it at any single run and it tells you what to fix, or what to keep. That is the self-improving loop the team actually asked for.

One run in

One run, no baseline.

learn <run_id> reads a single run — pass or fail. No paired A/B, no re-dispatching jobs just to learn something. A retro is one command.

Grounded

Grounded, not guessed.

A deterministic harvest builds phase parity and a failure taxonomy before any model runs. The one LLM step reads the digest, never the raw logs, and maps every fix to a finding id.

And it compounds

Compounds over runs.

New failure signatures fold back into the taxonomy as rules and tests, so the loop reads the next run more sharply. It doesn’t just report — it gets better.

Why a loop, not a log-read

The precision numbers are real: a naive full-log scan of the green run 28437323962 produced 10/10 false positives — echoed shell source, prompt text, the workflow’s own error-detectors. Scoping failures to --log-failed gave 100% precision — every flagged line a true positive. That one rule is the difference between a loop you trust and one you mute.

01How the loop works

learn is a two-part machine: a deterministic harvest that no model touches, followed by a single, tightly-scoped LLM step. Keeping the classification deterministic is what makes the output trustworthy.

1 · Deterministic harvest (no LLM)

It reads the run with gh and derives structure first: the run conclusion and its metric window; phase parity (the worst step per phase across smoke → design → pattern → deploy → invoke → legs → teardown); and the logs with precision — failing runs scoped to --log-failed, green runs scanned warnings-only. Anomalies are classified, deduped by category with a count and first evidence line, and written as a learnings/v1 digest.

2 · A reality-tuned taxonomy (ordered, first match wins)

Anomalies drop into a fixed, ordered taxonomy. Order is deliberate: rate_limit sits ahead of model_unavailable so “exceeded rate limit” can’t be miscounted as a transient error.

dependency› skill_loader› wire_protocol› protocol_contract› rate_limit› model_unavailable› auth› quota› deploy› tool_failure› router_fallback› retry› slow_turn

Highlighted classes are high severity. Green runs emit none of them — at most a low-sev retry or slow-turn surfaces, which is exactly how a healthy run should read.

3 · The single LLM step: digest → fixes

Only after the digest is written does a model get involved — and it reads the digest, never the raw logs. For each finding it names the most likely root cause and the smallest durable fix (a pin, a backoff, a quota bump, a wire-API switch, an RBAC grant), and maps the recommendation back to a finding id. If the run is green with zero findings, it says so — it is forbidden from inventing issues that aren’t in the digest.

02What it caught — and taught itself

The loop earned its keep on this very exercise. Pointed at the failing runs, it turned logs into fixes with no guesswork and no re-running pairs:

  • A missing skill dependency (a skill_loader finding) quietly stalling the hard workload before the agent reached its later phases.
  • A wire-protocol mismatch (a wire_protocol finding) surfaced by a Phase-4 protocol probe added to the workload pack.

Both fixes shipped straight from those digests. And the more durable win: the taxonomy itself got sharper.

  • Added a protocol_contract rule so wire-contract mismatches are caught by category, not by chance.
  • Promoted skill_loader from medium to high severity — a missing skill is a hard blocker, not a warning.
  • The skill’s own test suite grew 51 → 53 to lock the new behavior in.

03One thing the loop found: routing can right-size cost

Because we could learn from any run, we could afford to ask a bigger, optional question: is a model-router actually more efficient than a fixed cheap model? The heavier bench and validate paths are built to answer it — a controlled matrix through the same threadlight-e2e pipeline: a cheap baseline (gpt-5.4-mini), the router constrained to {gpt-5.4, gpt-5.4-mini}, and a quality ceiling (gpt-5.4), across a simple and a hard workload, sketched on rounds and modeled Azure spend.

Run the loop yourself

  1. Pick any run. Green or red, one run id. No pair, no baseline.
  2. Harvest it. learn <run_id> builds phase parity and the taxonomy digest — deterministic, no model in the loop yet.
  3. Read the digest, not the logs. The one LLM step turns findings into ranked fixes, each mapped to a finding id.
  4. Ship the fix. The smallest durable change per finding — a pin, a backoff, a wire-API switch.
  5. Fold it back. A recurring signature becomes a new rule and test, so the loop reads the next run more sharply.
