The run

Fourteen stages. Every one tested, every one reviewed.

The spine of the case study. Each stage ran a real Threadlight skill, was gated, and was independently reviewed — and each surfaced honest dogfooding findings sent back upstream without ever editing the skill in flight.

1

Design

threadlight-design~23 minPass

Design — one paragraph becomes a 958-line spec

The skill invented a coherent, internally consistent world: 12 business rules with real banking thresholds (DSCR ≥ 1.25×, leverage ≤ 3.5×, current ratio ≥ 1.20×, a $5M single-borrower limit, mandatory policy citation, no self-approve, mandatory human signoff), six agent skills, and a seeded demo world whose metrics actually compute.

Standout The numbers are correct, not just plausible — recomputing from seed data reproduces the spec’s headline ratios to two decimals, and the eval set agrees. Zero [NEEDS CLARIFICATION] markers.

2

Local test

threadlight-local-testPass

Local test — run it before you deploy it

Before spending a minute on Azure, the same agent code boots on a laptop against a real LLM — no Docker, no azd. The full six-skill agent reproduced every golden ratio exactly, with a policy citation on every assessed line and no self-approval.

Dogfooding finding Surfaced and fixed two genuine bugs in the quickstart harness itself — a tool-name collision that crashed agent construction, and a stale Azure OpenAI client class — caught only because local-test actually builds the agent.

3

Deploy · MVP

threadlight-deployazd up -e devPass · live

Deploy — the idea is now live on Azure

A single azd up turned the project into a containerized Microsoft Foundry hosted agent. Invoking the live agent reproduced both golden cases exactly — and left the memo at pending_review, because a human signs off, not the agent.

Recovered Hit two real failure modes and recovered from both: a post-deploy name-lookup 404 (one key renamed in azure.yaml) and the skipped role assignments it caused (granted and re-invoked). A regional quota crunch forced a same-family model swap — never a gpt-4.

4

Safety gate

threadlight-safe-checkConditional

Safe-check — the gate that tells the truth

The mandatory completeness gate came back red on all three phases — and that was the most useful result so far. Of 27 reported gaps, 26 were the hardening pieces we deliberately deferred to production, and exactly one was a real (trivial) defect: a workspace missing from the manifest.

Two findings A shell-quoting bug made the gate unrunnable on macOS/Linux — found, fixed, verified. And a design insight: the pipeline needs per-environment targets so a deliberately-lean pilot isn’t judged against a full-production manifest.

5

Cost

threadlight-consumption-iq17.5 minPass

What will this cost? — and the report that corrected itself

The first headline was $39,343/month. That number is wrong — and the skill’s own report said so. 99% was the model, priced as if it ran at peak request rate every second. The agent caught it, recomputed against realistic volume, and published ~$920/mo zone-redundant (or ~$125 with a documented exception).

The real insight The language model is a rounding error (~$11/mo). What drives the bill is whether the policy search index runs zone-redundant (~$736) or single-replica (~$74) — a business decision, now backed by real numbers.

6

Evaluate

foundry-evals30 minPass

Does the agent actually work? — 14 / 15

A real evaluation against the live agent, all fifteen scenarios scored by deterministic graders and a gpt-5 judge. It caught 100% of seeded policy exceptions, cited a clause on 41 of 41 measured lines, never self-finalized a memo, and earned 4.9/5 for coherence. Each memo: ~20s versus 3–4 human hours.

The instructive miss The single failure was a gap in the test, not the agent — a scenario with no borrower, where the agent correctly refused to invent an identifier. A finding also corrected an over-broad doc claim that gpt-5.4 was unusable as a judge.

7

Observe

foundry-observability21 minPass

Can we see what it’s doing?

The first queries came back empty — and untangling that was the stage’s most useful work. Pointed at the right tables, the picture lit up: ~2,000 trace records, every one of the fifteen eval runs accounted for, each model call and internal tool call traced as its own span, and a second independent confirmation the agent runs on gpt-5.

Send this upstream first The stock diagnostic queries return zero rows on a perfectly healthy agent — they target the wrong tables and assume role names this runtime no longer uses. The check meant to catch silent failures was itself failing silently.

Live demo

azd ai agent invoke ×2MVP capstone

The demo — one clean loan, one that sets off every alarm

Two cases, run live from the command line. The clean borrower came back in ~30s as a polished memo, every metric passing and cited, routed for sign-off, pending review. The hard borrower tripped three policy floors at once — each flagged with threshold, value, clause, and mitigants — and was escalated to a senior credit officer.

Written into the prep guide The second demo came back empty the first time — the CLI silently reused the first conversation’s context. A clean-conversation flag fixed it: the exact small thing that derails a live customer demo.

Readiness

threadlight-production-ready18.7 minPass

Is it ready? — not ready, 45%, and that’s correct

Scored against the bank’s declared target — a private, gateway-fronted spoke — the lean pilot came back not ready, 45%. The right answer. Crucially, the report pinned each of 34 gaps to the exact later stage that closes it, rather than just listing them.

Grade the assembly line too The scorecard tool fought us — two crashes on modern infra, an action plan empty of its own findings, a misread spec. The agent root-caused each to the line and wrote up 12 concrete bug reports (5 serious).

10

Architecture

operatorArchitecture review

Drawing the production map

We probed the bank’s Citadel hub live: a managed gateway, private network, multi-region control plane, caching, metering. One detail reshaped the plan — the gateway is reachable over the public internet with private backends, so we could onboard the light way: a keyless managed-identity connection plus a small isolated product, no VNet peering into shared infrastructure.

Caught early The shared hub still offers a deprecated model to anyone who asks — exactly what this project bans. We can’t remove it, but we can wall the spoke off: an allow-list naming only the two approved current-generation models.

11

Govern · spoke

citadel-spoke-onboarding13.8 minPass

Building the spoke — and proving the wall holds

Onboarding registered a small isolated product as an infrastructure-as-code contract — entirely additive (preview: 6 to create, 0 to modify, 0 to delete; the hub still listed all 23 existing products afterward). Then we turned the claim into a fact: approved model → 200 with the gateway’s own record of which version answered; banned deprecated model → 403, refused outright.

Priority fix The sample onboarding policy ships an allow-list that includes the very deprecated models the bank has banned. A team copying the default would have onboarded with the wrong rule baked in — the exact class of mistake running it for real is meant to catch.

12

Ship · prod

threadlight-deploy · prod36.2 minPass

Production, routed through the gateway — and proven

The production agent deploys no model of its own — it borrows the hub’s current-generation model through the gateway. That is the governed posture and it consumed zero new model quota. We ran both golden cases against live prod; both matched to the decimal. Then we checked the hub’s own logs: our calls, attributed to our product, succeeding — the banned model refused right alongside.

The run’s headline finding The playbook’s example pointed the connection at the wrong gateway surface — the agent returned empty, a silent “resource not found.” The real shape is an OpenAI-style endpoint with the model in the body (/models, not /openai). Re-pointed, redeployed, alive. The worst kind of failure: silent.

13

CI / CD

threadlight-cicd5.8 minPass

Handing the keys to a pipeline, not a person

No human and no agent holds standing deploy rights — a pipeline does, under a machine identity with no password at all. Federated OIDC tokens, a human approval gate before prod, provision and deploy as two auditable steps. We created the identity and proved the scoping: a tenant-wide listing returned exactly two permissions, both confined to the prod resource group — nothing at the subscription level, nothing near the shared gateway.

A seam between two tools The pipeline generator produces a correct, secret-free workflow — but doesn’t yet know the gateway-routed model wiring, so the workflow needs a small hand-edit to reproduce the Stage-12 setup. Only visible by running the whole chain.

14

Handoff

threadlight-customize9.9 minPass

Handing it to a real customer

The last stage ships runbooks and fill-in templates, not a generator — no two customers share a network or a compliance posture. The centerpiece is a customization map: every tool marked “keep” or “override, and here’s the knob,” each override citing a real moment from this run. The customer profile carries 20+ confirm-with-customer markers — honest about what was assumed for the demo.

Fitting, for a stage about honesty Running the templates for real exposed that the generic map didn’t anticipate the one override that defined the whole production leg — borrowing the model from a shared gateway instead of deploying your own. Logged, with seven more.

Act IV

It reached production. It is not yet trustworthy everywhere.

A report you can trust is one that’s honest about where it can’t be trusted. The run shipped nearly 60 findings upstream — including defects that cut against the project’s own rules — and a clear statement of what Threadlight refuses to automate at all.

The adverse review

Where it can’t be trusted — yet.

This is the section a vendor deck would delete. Four defects the run hit head-on, stated plainly — two of them cut directly against the project’s own hard rule.

Self-improvingEvery issue here — and the ~54 lower-severity findings alongside them — was logged in the run repo with a root cause, then fed straight back as upstream fixes to the skills themselves. Dog-fooding the pipeline is how the pipeline gets better. ~58 findings · logged & routed upstream · none blocked production

High · against the hard rule

gpt-4 ships in the defaults

The accelerator’s default gateway policy allow-lists deprecated gpt-4-class models — allowedModels="gpt-4o,…,gpt-4.1,…". A spoke that copies the default silently permits the exact models the brief forbade. We caught it because we read the policy; a hurried operator wouldn’t.

High · silent failure

The 404 routing trap

Hosted-agent gateway routing needs the OpenAI-style /models body surface, but the shipped example points at the path-based /openai surface. The result isn’t an error — it’s empty output and a transient 404 on the poll path. The costliest hour of the run.

Medium · blind telemetry

KQL that sees zero rows

The stock observability queries assume a classic schema and role names this Foundry runtime no longer emits. On a perfectly healthy agent they return nothing — the telemetry is flowing; the queries simply can’t find it. A false “it’s broken” waiting to happen.

Medium · the cost headline

A number that’s 40× wrong

Left unread, the cost skill’s $39,343/mo headline would have sunk the business case on its own. It was only trustworthy because the agent flagged its own number — but the skill still emitted it as the top line. The price model can’t yet quote gpt-5 natively.

Drawing the line

It automated the pilot. It refuses to automate your platform.

The clearest way to read the run: everything on the left, Threadlight built and proved on its own. Everything on the right it deliberately hands to your platform team — and naming that boundary out loud is itself the deliverable.
