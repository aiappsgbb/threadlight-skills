Stages

Threadlight  /  The funnel

Five named stages · one session

From a paragraph to a governed agent — in five named stages.

Every Threadlight engagement walks the same five-stage funnel. Each stage has its own skills, its own artefacts, and its own exit criteria. The skills do the heavy lifting; the seller stays in control of the conversation.

Stage 01

Conversation — describe the process.

A plain-language brief from the seller or customer SME. No template, no questionnaire. Just enough fidelity that a coding agent can start drafting a spec.

What ships

  • Conversation transcript (markdown)
  • Draft problem statement
  • Inferred actors + data sources
  • First-cut success criteria

Skills used

threadlight-auto · threadlight-design

The first hands the seller a guided structure; the second turns the conversation into a versioned SPEC.

Exit criteria

  • Process scope is named in one paragraph
  • Inputs / outputs are explicit
  • Regulator / data class is acknowledged

What it is not

It is not a discovery workshop. The seller does not need a slideware deck before the customer sees the agent run.

Stage 02

Co-design — spec & scaffold.

The brief becomes a working agent on the seller's laptop. SPEC.md, AGENTS.md, an eval dataset, and a demo-data factory — generated in-session by the coding agent.

What ships

  • SPEC.md — versioned process spec
  • AGENTS.md — agent contract
  • Agent code scaffold (Python/Node)
  • Synthetic demo data + eval dataset

Skills used

threadlight-design · threadlight-demo-data-factory

Design owns the spec/scaffold loop. Demo-data-factory invents customer-shaped data so the agent runs from minute one.

Exit criteria

  • Agent runs locally against demo data
  • Spec passes its own self-review
  • Eval cases cover the happy path + 1 HITL gate

Why it matters

The customer sees the agent move, in their domain, before they have to commit to architecture or budget.

Stage 03

Deploy — land on your Azure.

Bicep + azd up to the customer's subscription. Managed Identity, App Insights, and the AI Citadel gateway wired from day zero. No service principals; no shared secrets.

What ships

  • Bicep modules per Azure resource
  • azure.yaml + azd environment
  • Dockerfile + Container Apps job
  • App Insights + Log Analytics wired

Skills used

threadlight-deploy · threadlight-local-test

Local-test catches drift before the cloud bill arrives. Deploy walks the agent from az login to a healthy endpoint.

Exit criteria

  • Agent responds in the customer's tenant
  • App Insights shows the request trace
  • All identities are managed, not shared

Day-zero invariants

AI Citadel APIM gateway on the perimeter; AGT middleware inside the agent. The CISO reviews Bicep before anything ships.

Stage 04

Safe-check — cost & readiness.

Project a credible monthly cost for the production load profile. Recommend cheaper SKUs where the headroom is real. Score the agent against 10 production pillars.

What ships

  • Cost projection by SKU + region
  • Cheaper-SKU recommendations with the maths
  • 10-pillar readiness scorecard
  • Remediation backlog with owners

Skills used

threadlight-safe-check · threadlight-consumption-iq

Consumption-IQ does the cost arithmetic across Azure SKUs. Safe-check is the soft-gate to production.

Exit criteria

  • Cost number the seller will defend to the CFO
  • Top 3 readiness gaps named with owners
  • No critical-pillar regressions vs the baseline

Why a soft-gate

The customer decides whether to ship. The agent shows them exactly what they are signing up for — in their numbers, not a Microsoft list price.

Stage 05

Production — hand-off.

Governance, telemetry, and runbooks already in the repo. HITL gates already wired. The customer keeps the agent — and the repo — and can change it without us.

What ships

  • Production runbook + on-call rota template
  • Eval pipeline wired into CI
  • SRE Agent hook-ups (optional)
  • Customer architecture-review pack

Skills used

threadlight-production-ready · threadlight-hitl-patterns

Production-ready scores the agent across 13 pillars and produces the hand-off report. HITL-patterns documents the escalation surface.

Exit criteria

  • Customer can deploy + roll back without us
  • Telemetry-driven alerts wired and tested
  • Architecture-review hand-off accepted

What hand-off means

Threadlight is not a managed service. The customer owns the agent, the repo, and the Azure subscription. We make sure they can keep going on Monday.

One funnel. Five exit gates.

Each stage has a named exit. Each exit produces evidence. When a stage closes, the seller and the customer agree on what just shipped — before the next stage starts.
