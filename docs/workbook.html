Before you start

Six things on the bench, then one pre-flight prompt.

Everything signed in before Step 1 — not discovered mid-runbook. The skills authenticate with Entra (DefaultAzureCredential), so there are no API keys to paste anywhere.

Copilot CLI

The primary surface. Run it on a current-generation model — this workbook assumes claude-opus-4.8 (default) or gpt-5.5. A stronger agent writes a stronger spec and catches its own bugs.

GitHub CLI · gh

Signed in. Installs the two plugins and, in the fallback path, reaches GitHub Models for the local step.

Azure CLI · az

Signed in to a subscription you can deploy to. Entra auth, no keys. Part 1 runs entirely here.

Python 3.13

The local-test harness pins it. If you’re older, the recovery prompts spin up a uv 3.13 venv in one line.

A model to call

An Azure OpenAI deployment you can reach (endpoint + deployment name), or accept the GitHub Models fallback for the local step.

Git + an editor

A clean working folder to git init the project into. You’ll read the generated spec and re-derive the numbers yourself.

One prompt confirms the bench before you start the runbook:

copilot — pre-flight
$ copilot verify python 3.13, gh, az and copilot CLI are installed and signed in, and tell me which subscription az is pointed at
Heads-up · vocabulary The skills carry labels from the team that wrote them — GBB, seller/SE, pilot/pre-pilot, prep guide. That’s flavour text in the output, not configuration you fill in. The engineering they describe — spec, build, deploy, govern — is exactly what you’re doing.

Part 1

Hands-on to a working MVP.

Five steps, each one prompt into Copilot CLI. After every step, a “you’ll know it worked when” checkpoint — so you can tell green from broken without guessing.

1

install + launch

Install the plugins, launch on a strong model

Pull both skill libraries from the aiappsgbb org, then start Copilot CLI on a current-generation model so the inner agent is strong enough to spec and self-correct.

copilot — install plugins
$ copilot install the awesome-gbb and threadlight-skills plugins from the aiappsgbb org on github, then list my installed copilot plugins
shell — launch on a strong model
$ copilot --model claude-opus-4.8 --effort high \ --plugin-dir ./threadlight-skills --plugin-dir ./awesome-gbb

You'll know it worked when

The plugin list shows both threadlight-skills and awesome-gbb, and Copilot reports the active model is claude-opus-4.8 (or gpt-5.5).

2

threadlight-design

Design — your paragraph becomes a spec

The verbatim prompt that started the real run. Swap the BRIEF for your own process and keep the rest — every later prompt stays the same.

copilot — the brief (credit-memo example)
› Use the threadlight-design skill to design a pilot agent end-to-end from this brief. BRIEF A commercial bank's SMB lending team wants to compress credit-memo preparation. For an incoming loan request, the agent should: (1) pull the borrower's financials and existing exposure, (2) compute standard credit metrics (DSCR, leverage, liquidity) and score the request against the bank's lending policy, (3) flag policy exceptions and risk factors, and (4) draft a structured credit memo a credit officer reviews and signs. A human-in-the-loop approval gate is required before any memo is finalized, and every step must be auditable. CONTEXT - Fictional customer "Meridian Commercial Bank", SMB lending. - This is a PILOT we will take to production on Azure later in the same session, onboarded as a spoke into an existing Citadel AI governance hub. Design with that path in mind (governance, auditability, PII, HITL) but keep the pilot itself lean. - Region: swedencentral. MODEL POLICY: use a current-generation model (gpt-5 family). Production will route LLM calls through the Citadel AI gateway. Produce the full output set: specs/SPEC.md, specs/manifest.json, AGENTS.md, the per-tool skills/ scaffold, and docs/.

You'll know it worked when

specs/SPEC.md exists with concrete business rules — the credit-memo brief yields ~12 with real thresholds (DSCR ≥ 1.25×, leverage ≤ 3.5×, a single-borrower limit, mandatory policy citation, no self-approve) — plus AGENTS.md, agent skill folders, and seeded specs/sample-data/. Zero [NEEDS CLARIFICATION] markers is the bar.

Re-derive it yourself: open the sample data and recompute one headline ratio by hand. In the real run the numbers were correct, not just plausible — they reproduced to two decimals. That’s the test.

If it drifts Short on time and it starts generating a demo deck or sales kit? Interrupt with skip the demo and sales kit, move on to protect budget.
3

threadlight-local-test

Local test — run it before you deploy it

Prove the generated agent actually boots and produces correct answers on a laptop, before spending a minute on Azure.

copilot — Pattern 0 smoke test
› Use the threadlight-local-test skill to boot this PoC locally (Pattern 0 — Quickstart) and run a real smoke test on the two golden demo cases. Prefer GitHub Models (LLM_BACKEND=copilot, auth via gh) with a gpt-5-family model; if none is offered, use an Azure OpenAI gpt-5-family deployment. Then run --info and --check.

You'll know it worked when

--info prints the auto-discovered entities and CRUD tools, --check exits clean, and the smoke test reproduces the golden ratios exactly — a policy citation on every assessed line, no self-approval.

Why this step earns its keep In the real run this is where two genuine bugs surfaced — in the harness itself — because it actually builds the agent rather than describing it.
4

threadlight-deploy · azd up -e dev

Deploy — the idea goes live on your sub

One azd up turns the project into a containerized Microsoft Foundry hosted agent on your subscription. Keep the Azure Portal resource-group view open in a second tab — you’ll watch it fill in live.

copilot — deploy to your subscription
› Use the threadlight-deploy skill to azd up this project to Azure as a dev environment (-e dev) on my current subscription, region swedencentral.

You'll know it worked when

azd up finishes green and the resource group shows a Foundry account + project, a Container App, and the hosted-agent container. This is the longest single step (~15–25 min, region/cache dependent).

Real failure modes here The run hit — and recovered from — a post-deploy 404 name-lookup (one key renamed in azure.yaml) and the skipped role assignments it caused. If you see either, the recovery prompts have the one-line fix.
5

azd ai agent invoke × 2

Invoke — the two cases that prove it

Run the agent live, from the command line, on two contrasting cases — one clean, one that trips every alarm.

copilot — invoke the live agent
› Use azd ai agent invoke to run the live agent on the first golden case (the clean borrower). Then invoke it again on the hard borrower — start a CLEAN conversation for the second so it doesn't reuse the first one's context. Show me both memos and where each was routed.

You'll know it worked when

The clean borrower returns a polished memo, every metric passing and cited, left at pending_review — a human signs off, not the agent. The hard borrower trips its policy floors (each flagged with threshold, value, and clause) and escalates.

That pending_review state is the whole point: the agent drafts, a person decides.

🎉 That’s the MVP You took a paragraph to a governed-ready agent on your own Azure — and proved it yourself. Everything past here is the road to production.

Part 2

The production track.

The real prompts from the run, in order. Reproducible — but several stages call a Citadel governance hub you probably don’t own, and the track spends real Azure money over a few hours. This is the documented path; the case study is the evidence it lands.

🟢 your own sub 🟡 needs time + real spend 🔵 needs a Citadel hub you can onboard to
6

Safe-check — the gate that tells the truth

threadlight-safe-check
🟡 time + spend
copilot
› Use the threadlight-safe-check skill to run all phases against this project and tell me, gap by gap, what's a real defect versus deliberately-deferred hardening.

In the run it came back red on all three phases — and that was the most useful result yet: of 27 gaps, 26 were hardening deliberately deferred to production, exactly one a real (trivial) defect. A red safe-check on a lean pilot is correct, not a failure.

7

Cost — what will this actually cost?

threadlight-consumption-iq
🟡 time + spend
copilot
› Use the threadlight-consumption-iq skill to forecast the monthly run cost of this agent at realistic request volume, and break the bill down by resource.

The first headline was $39,343/mo — and wrong (priced as if it ran flat-out every second). The skill’s own report caught it and recomputed to ~$920/mo zone-redundant (or ~$125 with a documented exception). Worth re-deriving: the language model is a rounding error (~$11/mo) — the bill is the policy search index.

8

Evals — does it actually work?

foundry-evals
🟡 time + spend
copilot
› Use the foundry-evals skill to evaluate the live agent across the full scenario set, scored by deterministic graders and a gpt-5 judge. Report exception recall, citation coverage, and whether it ever self-finalized.

The run scored 14/15: 100% of seeded exceptions caught, a clause cited on 41/41 measured lines, never self-finalized, 4.9/5 coherence — each memo ~20s versus 3–4 human hours. The single “miss” was a gap in the test, not the agent.

9

Observability — can we see what it’s doing?

foundry-observability
🟡 time + spend
copilot
› Use the foundry-observability skill to confirm the agent is emitting traces, and show me the spans for the eval runs — model calls and tool calls.

Pointed at the right tables: ~2,000 trace records, every eval run accounted for, each model and tool call its own span.

Finding The stock diagnostic queries returned zero rows on a perfectly healthy agent — wrong tables. If your first queries come back empty, the agent isn’t necessarily broken; the query might be.
10

Production-ready — is it ready?

threadlight-production-ready
🟡 time + spend
copilot
› Use the threadlight-production-ready skill to score this pilot against a private, gateway-fronted spoke target, and pin every gap to the stage that closes it.

Scored against the bank’s declared target, the lean pilot came back not ready, 45% — the right answer. The valuable part: each of 34 gaps pinned to the exact later stage that closes it, rather than just listed. A 45% here is a correct read of a deliberately-lean pilot.

11

Citadel spoke — onboard behind the gateway

citadel-spoke-onboarding
🔵 needs a Citadel hub🟡 time + spend
copilot — probe the hub, then onboard
› Probe the target Citadel hub: is the AI gateway reachable over the public internet with private backends, or VNet-only? List which models it exposes and flag any deprecated ones. › Use the citadel-spoke-onboarding skill to register this agent as an isolated spoke product on the hub — keyless managed-identity connection, an allow-list naming ONLY the current-generation models we approve, and preview the change before applying. It must be purely additive: nothing modified or deleted on the shared hub.

Entirely additive in the run (6 to create, 0 to modify, 0 to delete; the hub still listed all 23 existing products). Then the wall was proven: approved model → 200; banned deprecated model → 403, refused.

Carry this in The sample onboarding policy ships an allow-list that includes deprecated models. Copy the default blindly and you onboard with the wrong rule baked in. Name your approved models explicitly.
Private networking Here the spoke reaches the hub’s public gateway (TLS · JWT · allow-list) while the hub’s model backends sit private behind private endpoints — the shipped, additive-only posture. A fully private VNet-peered spoke (spoke VNet ↔ hub VNet + Private Link) is a first-class, skill-covered path — foundry-vnet-deploy plus citadel-spoke-onboarding’s VNet-isolated (Option B) pattern — when the bank is ready to peer.
12

Production deploy — routed through the gateway

threadlight-deploy · prod
🔵 needs the onboarded spoke🟡 time + spend
copilot
› Use the threadlight-deploy skill to deploy this agent to a prod environment that borrows the hub's current-generation model THROUGH the Citadel gateway — no model of its own. Then run both golden cases against live prod and check the hub's own logs to confirm our calls are attributed to our product.

The production agent deploys no model of its own — it borrows the hub’s through the gateway. That’s the governed posture and it consumes zero new model quota. Both golden cases matched to the decimal against live prod.

The run’s headline finding The playbook’s example pointed the connection at the wrong gateway surface and the agent returned empty — a silent “resource not found.” The real shape is an OpenAI-style endpoint with the model in the body (/models, not /openai). Re-pointed, redeployed, alive.
13

CI/CD — hand the keys to a pipeline

threadlight-cicd
🟡 time + spend
copilot
› Use the threadlight-cicd skill to generate a deploy pipeline under a machine identity with federated OIDC (no passwords), a human approval gate before prod, and provision/deploy as two auditable steps. Then prove the identity's permissions are scoped to ONLY the prod resource group.

No human and no agent holds standing deploy rights — a pipeline does, under a passwordless machine identity. A tenant-wide listing returned exactly two permissions, both confined to the prod resource group; nothing at the subscription level, nothing near the shared gateway.

Finding The generated workflow is correct and secret-free, but doesn’t yet know the gateway-routed model wiring from Stage 12 — it needs a small hand-edit to reproduce that. Only visible by running the whole chain.
