Three postures · thirteen pillars · one scorecard

Green safe-check isn’t go-live.

A pilot that passes threadlight-safe-check --phase post-deploy is structurally complete: every selector landed, every channel reaches, no placeholder image. That is not the same as ready for the customer architecture review. This chapter is what we walk through with the customer to answer “what posture is this in, what’s missing, who owns each gap, can we go live with waivers?”

threadlight-production-ready produces the answer in one command — soft-advisory (never fails a build), graceful when Azure permissions are partial (degrades to not-verified, never invents a green), and posture-aware (it doesn’t recommend Citadel if the customer is on a brownfield gateway).

When to graduate

Six questions a CISO asks. If any one fires, you’ve outgrown the laptop.

Threadlight skills run cleanly from a developer’s machine for early-stage co-design. The moment any of these six triggers turns red, the pilot needs a hosted runtime, governed identity, and a paved-path posture — not another iteration on the laptop. This is the conversation production-ready opens with — and, for governance, evals, and safety, the conversation is now backed by legs that actually run.

01 · Identity

Who is the agent acting as?

If you can’t answer in one sentence — UAMI, Entra app, on-behalf-of — you’re past laptop scope.

02 · Audit

What did it do, when, for whom?

Append-only audit and OTel traces are the difference between a demo and a defensible deployment.

03 · RBAC

Can it read this index / container / API?

Keyless RBAC per service, scoped to the agent identity — not a developer’s personal token.

04 · Telemetry

How do we know it’s running well?

App Insights + OTel from day one, not bolted on after the first incident.

05 · Evals

How do we know the next deploy didn’t regress?

Foundry Evals as a continuous gate — quality is a release criterion, not a vibe-check.

06 · Safety

Can it be jailbroken or made to exfiltrate?

An adversarial scan — prompt-injection, jailbreak, data and prompt exfiltration — run before and after deploy, not assumed because a shield was declared. This is the red-team leg.

Discover → Protect → Govern

Three of those questions are now legs that run — not slides.

Microsoft frames the agent lifecycle as Discover → Protect → Govern: adversarially test to discover, wire model and agent-runtime controls to protect, monitor and evaluate on live traffic to govern. Production-ready used to score those stages and hand the work to other skills. Now several threadlight legs run as executable checks in the chain and leave committed evidence the scorecard reads — governance, evals, red-team, the CI gates, and an MCP supply-chain SBOM — idempotent, keyless, and gracefully degrading to not-verified when a permission is missing.

design deploy discover · evals + red-team protect · AGT govern · online eval + KPIs improve

Protect · threadlight-govern

Agent-runtime governance (AGT) that ships an artefact.

Wraps foundry-agt to scaffold and validate a policy artefact, wire in-process middleware at the container boundary, and emit a committed verifier report — the evidence pillars 02 (Agent governance) and 07 (Responsible AI) look for, instead of a remediation pointer.

Discover + Govern · threadlight-evals

Offline batch, online continuous eval, and an A/B gate.

Runs the offline quality batch, wires Foundry Continuous Evaluation on live threads (results to App Insights), and gates model or prompt swaps behind an A/B champion–challenger comparison. Emits specs/evals-manifest.json whose metrics.pass_rate is the join key the outcome-KPI scorecard reads.

Discover · threadlight-redteam

An adversarial scan, not a declared shield.

Ingests and scores the AI Red Teaming Agent (PyRIT-based) scan evidence — jailbreak, prompt-injection, data and prompt exfiltration — into an attack-success-rate verdict, and emits docs/redteam-report.md plus a manifest whose findings map to Responsible-AI pillar 07. Run the scan pre- and post-deploy; a declared jailbreak shield is no longer taken on faith.

Govern · threadlight-cicd

Eval, red-team, and MCP gates in the pipeline.

The CAF process puts evaluation in CI/CD. The generated pipeline now carries an eval-gate, a red-team-gate, and an mcp-gate — soft (advisory) or hard (blocking) by a flag, OIDC, secret-free — so a regression, a fresh jailbreak, or an unpinned MCP server stops the deploy. See the pipeline below.

Browse the leg ↗

Discover · threadlight-production-ready

An SBOM for the agent’s MCP servers & tools.

Discovers every MCP server the agent mounts and emits mcp-sbom.json — pinned version + digest, resolved registry, a SHA-256 over each tool’s description and input schema — then diffs it against a committed mcp-lock.json so a silent server swap or a rug-pulled tool is caught. Pillar 09 reads the verdict (SUP-010–013) instead of a remediation pointer; the --mcp-gate blocks a deploy on an unpinned or drifted server.

Postures

Three shapes — resolved from SPEC § 12.

The right answer for “is this production-ready?” depends on the customer’s estate. We support three postures explicitly, and threadlight-production-ready resolves which one applies before it starts asking governance questions.

01 · Gateway-fronted (Citadel spoke)DEFAULT

Gateway-fronted — the Citadel spoke.

Customer’s AI Citadel hub fronts every model call via APIM (JWT, rate-limit, content filter, semantic cache, ledger). Pilot Foundry project is onboarded as a spoke; agent-governance-toolkit enforces fine-grained policy inside the agent itself.

2 layers pillars 01 · 03 · 04 · 09 pillars 02 · 05 · 07

02 · Foundry-native (core only)

Foundry-native — core only.

No central AI gateway. Agent Governance Toolkit (v4.1 preview, detected automatically) sits inside the agent process. 8–12 µs/eval, hash-chained audit, OWASP-ASI evidence. Greenfield or experimental tenants where the gateway investment hasn’t happened.

1 layer pillars 03 · 04 · 07 · 11

03 · Customer-owned edge

Customer-owned edge.

Brownfield or regulated estate with existing APIM, NetSec-controlled VNet injection, or Defender for Cloud baseline. Pilot conforms to the established gateway pattern rather than introducing Citadel mid-flight. The safe-check verifies, never prescribes.

How we pick

The SPEC’s § 12 (governance posture) is the source of truth. If the customer hasn’t declared one, threadlight-design asks the question during co-design — explicitly, with the trade-offs — rather than picking silently. If detection finds a Citadel-pattern APIM–Foundry binding, posture 01 resolves automatically; an APIM binding outside the Citadel pattern resolves to posture 03 (customer-owned edge) regardless of what the SPEC says.

The outcome, not just the shape

One more page — does it actually pay off?

The 13-pillar scorecard proves the pilot is production-shaped. It doesn’t prove it’s working. The CAF agent-observability baseline is explicit: trace latency, cost-per-interaction, and success rate, set baselines, and alert on deviation. Production-ready now emits an outcome-KPI scorecard (report § 8) that joins the evals leg’s live pass_rate, consumption-iq’s cost-per-interaction, and the observability baselines into one measurable view — eval quality stops being a number locked inside the evals leg.

Pillar 10 · deep-dive

Forecast the bill at production load — before turning on traffic.

threadlight-consumption-iq walks every deployed resource, projects it to the customer’s declared production load, prices it against the public Azure Retail Prices API, and emits a manifest the ARB and FinOps can govern against. The number on the scorecard is a forecast, not an invoice surprise.

Three things this answers — (1) does the pilot SKU mix scale from PoC traffic to production load without breaking constraints?   (2) what’s the governance tag the customer’s FinOps needs (budget, owner, posture)?   (3) is every line traceable to a real Azure price we can re-run on demand?

After green · the handoff

Green scorecard — now ship it through a pipeline, not your laptop.

The scorecard proves the pilot is production-shaped. It does not prove that by running azd up from a laptop — in a real customer tenant the agent rarely has those rights. Prod deploys go through CI/CD, run by a federated identity with scoped RBAC, often from a private-VNet runner. threadlight-cicd generates that pipeline and the env-setup runbooks the platform team executes — it never assumes deploy rights of its own. It’s a deliberate manual handoff, not part of the one-session chain. And when the pilot becomes a customer’s own — forked, upstream-pinned, onboarded into their estate — that’s the final leg, threadlight-customize.

01 · Two platforms

Your CI of choice.

Ready-to-run azd-deploy-prod.yml (GitHub Actions) or azure-pipelines.yml (Azure DevOps) — gated on an environment approval, re-running production-ready post-deploy.

02 · Secret-free identity

OIDC / Workload Identity Federation.

A UAMI plus federated credentials — no AZURE_CREDENTIALS, no client secret in the repo. Runbooks and az scripts the platform team runs; a test asserts no secret ever ships in a template.

03 · Scoped RBAC

Least-privilege, spoke-RG only.

Role assignments scoped to the target resource group — the same Identity and RBAC questions the scorecard asks, answered in IaC. Private-VNet deployments wire managed or self-hosted runners.

04 · Parallel track

Separate from the central platform.

The pilot pipeline is its own repo and pipeline. It never deploys or edits the Citadel hub, shared APIM, or platform networking — those stay with citadel-hub-deploy. A spoke posture consumes the hub through an Access Contract.
