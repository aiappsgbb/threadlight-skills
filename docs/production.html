Take an agent pilot to production — with evidence a review board signs.

threadlight-production-ready is a Markdown skill your coding agent runs. It resolves the customer’s architecture posture, scores the pilot against thirteen production pillars, and writes one scorecard plus the committed evidence behind it. Soft-advisory (it never fails a build), keyless, and graceful when Azure access is partial — it degrades to not-verified, never invents a green.

Why this chapter exists

Green safe-check isn’t go-live.

A pilot that passes threadlight-safe-check is structurally complete — every selector landed, every channel reaches. That is not the same as ready for the customer architecture review. Here is what closes the gap, what you walk away with, and why it’s fast.

The risk it kills

“It demos” is not “it’s governed.”

An agent with a real identity, network posture, audit trail, eval gate and cost ceiling is defensible. Without them it’s a demo you can’t put in front of a CISO. This chapter names every gap — and who owns it.

What you walk away with

One scorecard, plus the evidence.

Thirteen pillars in three colours — green, amber-with-owner, not-verified — and a committed file behind each. The architecture-review board reads the PDF in seven minutes and signs the waivers.

Why it’s fast

It drives the platform you already own.

Foundry Evals, the AI Red Teaming Agent, the Agent Governance Toolkit, Entra identity, the Azure Retail Prices API — the skill wields them for you and leaves the evidence. You adopt Microsoft’s own controls without wiring each one by hand.

threadlight-production-ready · pillar 09

An SBOM for the agent’s MCP surface.

Discovers every MCP server the agent mounts — pinned version + digest, a SHA-256 over each tool’s schema — then diffs it so a silent server swap or a rug-pulled tool is caught.
Leaves: mcp-sbom.json + mcp-lock.json.

Forecast the bill at production load.

Walks every deployed resource, projects it to declared production load, and prices it against the Azure Retail Prices API — every line tagged price_source, re-runnable.
Leaves: specs/cost-manifest.json.

Eval, red-team & MCP gates in CI.

Generates an OIDC, secret-free pipeline carrying an eval-gate, a red-team-gate and an mcp-gate — soft or hard by a flag.

After green · the handoff

Now ship it through a pipeline — not your laptop.

A green scorecard doesn’t mean azd up from a laptop — in a real tenant the agent rarely has those rights. Prod deploys run through CI/CD under a federated identity with scoped RBAC, often from a private-VNet runner. threadlight-cicd generates that pipeline and the env-setup runbooks the platform team executes — a deliberate manual handoff.

01 · Two platforms

Your CI of choice.

Ready-to-run azd-deploy-prod.yml (GitHub Actions) or azure-pipelines.yml (Azure DevOps) — gated on an environment approval, re-running production-ready post-deploy.

02 · Secret-free identity

OIDC / Workload Identity Federation.

A UAMI plus federated credentials — no AZURE_CREDENTIALS, no client secret in the repo. A test asserts no secret ever ships in a template.

03 · Scoped RBAC

Least-privilege, spoke-RG only.

Role assignments scoped to the target resource group — the same Identity and RBAC questions the scorecard asks, answered in IaC. Private-VNet deployments wire managed or self-hosted runners.

04 · Parallel track

Separate from the central platform.

The pilot pipeline is its own repo. It never edits the Citadel hub, shared APIM, or platform networking — those stay with citadel-hub-deploy. A spoke consumes the hub through an Access Contract.

Run it on your pilot today

Getting started — four prompts, one scorecard.

You need a repo with your agent pilot, GitHub Copilot (IDE, CLI, or the chat panel) with the Threadlight skills available, and — optionally — read access to the pilot’s Azure subscription. Without Azure access the cloud pillars resolve to not-verified instead of green; nothing fails. You talk to the agent in plain English — it reads the SKILL.md and does the work.

Step 01 · Score the pilot

Point your agent at the skill.

One prompt resolves the posture and walks all thirteen pillars.

> "Use threadlight-production-ready on this pilot. Detect the posture; production load is 800 rpm."
— reads skills/threadlight-production-ready/SKILL.md, resolves posture, walks 13 pillars,
  writes docs/production-readiness-report.md · ready in ~7m

Step 02 · Close the amber gaps

Run the legs that leave evidence.

Each amber pillar has a leg that turns it green — and commits the artefact behind it.

> "Use threadlight-govern to scaffold and validate the agent policy artefact."
> "Use threadlight-evals to wire continuous evaluation and an A/B release gate."
> "Use threadlight-redteam to score the AI Red Teaming Agent scan into a verdict."
> "Use threadlight-consumption-iq to forecast the bill at production load."

Step 03 · Gate the pipeline

Generate the secret-free deploy.

An OIDC pipeline with eval, red-team and MCP gates — ready for the platform team.

> "Use threadlight-cicd to generate the OIDC deploy pipeline with eval, red-team and MCP gates."
— writes azd-deploy-prod.yml (or azure-pipelines.yml) + env-setup runbooks · no secret in the repo

Step 04 · Read it & ship

Sign the waivers, hand it over.

Open docs/production-readiness-report.md, sign any amber pillar’s waiver with its named owner, and hand the pipeline to the platform team. Everything the board needs is committed in the repo — the scorecard, the SBOM, the eval and red-team manifests, the cost forecast. Nothing lives in a console tab you have to screenshot.
