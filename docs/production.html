Green safe-check isn’t go-live.

A pilot that passes threadlight-safe-check --phase post-deploy is structurally complete: every selector landed, every channel reaches, no placeholder image. That is not the same as ready for the customer architecture review. This chapter is what we walk through with the customer to answer “what posture is this in, what’s missing, who owns each gap, can we go live with waivers?”

threadlight-production-ready produces the answer in one command — soft-advisory (never fails a build), graceful when Azure permissions are partial (degrades to not-verified, never invents a green), and posture-aware (it doesn’t recommend Citadel if the customer is on a brownfield gateway).

When to graduate

Five questions a CISO asks. If any one fires, you’ve outgrown the laptop.

Threadlight skills run cleanly from a developer’s machine for early-stage co-design. The moment any of these five triggers turns red, the pilot needs a hosted runtime, governed identity, and a paved-path posture — not another iteration on the laptop. This is the conversation production-ready opens with.

01 · Identity

Who is the agent acting as?

If you can’t answer in one sentence — UAMI, Entra app, on-behalf-of — you’re past laptop scope.

02 · Audit

What did it do, when, for whom?

Append-only audit and OTel traces are the difference between a demo and a defensible deployment.

03 · RBAC

Can it read this index / container / API?

Keyless RBAC per service, scoped to the agent identity — not a developer’s personal token.

04 · Telemetry

How do we know it’s running well?

App Insights + OTel from day one, not bolted on after the first incident.

05 · Evals

How do we know the next deploy didn’t regress?

Foundry Evals as a continuous gate — quality is a release criterion, not a vibe-check.

Postures

Three shapes — resolved from SPEC § 12.

The right answer for “is this production-ready?” depends on the customer’s estate. We support three postures explicitly, and threadlight-production-ready resolves which one applies before it starts asking governance questions.

01 · Gateway-fronted (Citadel spoke)DEFAULT

Gateway-fronted — the Citadel spoke.

Customer’s AI Citadel hub fronts every model call via APIM (JWT, rate-limit, content filter, semantic cache, ledger). Pilot Foundry project is onboarded as a spoke; agent-governance-toolkit enforces fine-grained policy inside the agent itself.

2 layers pillars 01 · 03 · 04 · 09 pillars 02 · 05 · 07

02 · Foundry-native (core only)

Foundry-native — core only.

No central AI gateway. Agent Governance Toolkit (v4.1 preview, detected automatically) sits inside the agent process. 8–12 µs/eval, hash-chained audit, OWASP-ASI evidence. Greenfield or experimental tenants where the gateway investment hasn’t happened.

1 layer pillars 03 · 04 · 07 · 11

03 · Customer-owned edge

Customer-owned edge.

Brownfield or regulated estate with existing APIM, NetSec-controlled VNet injection, or Defender for Cloud baseline. Pilot conforms to the established gateway pattern rather than introducing Citadel mid-flight. The safe-check verifies, never prescribes.

1 layer pillars 01 · 03 · 09 · 12 · 13

How we pick

The SPEC’s § 12 (governance posture) is the source of truth. If the customer hasn’t declared one, threadlight-design asks the question during co-design — explicitly, with the trade-offs — rather than picking silently. If detection finds a Citadel-pattern APIM–Foundry binding, posture 01 resolves automatically; an APIM binding outside the Citadel pattern resolves to posture 03 (customer-owned edge) regardless of what the SPEC says.

Pillar 10 · deep-dive

Forecast the bill at production load — before turning on traffic.

threadlight-consumption-iq walks every deployed resource, projects it to the customer’s declared production load, prices it against the public Azure Retail Prices API, and emits a manifest the ARB and FinOps can govern against. The number on the scorecard is a forecast, not an invoice surprise.

Three things this answers — (1) does the pilot SKU mix scale from PoC traffic to production load without breaking constraints?   (2) what’s the governance tag the customer’s FinOps needs (budget, owner, posture)?   (3) is every line traceable to a real Azure price we can re-run on demand?

Recap
