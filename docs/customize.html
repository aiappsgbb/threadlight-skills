Chapter

Threadlight / Customize

From a working pilot to their production

threadlight-customize is the engagement runbook that takes a Threadlight pilot and fits it to one customer’s real production environment — by forking the skills, capturing their landing zone, identity, network and governance in an overlay, and onboarding without ever editing a skill file.

Structured intake

A customer-profile.md captures their environment, requirements and mandated templates — so nothing about production is guessed.

Overlay, not fork-edit

Every customer value lives in overlay/ beside the untouched skills, so git merge upstream/main keeps pulling our fixes.

A signed-off boundary

A non-coverage.md states what Threadlight deliberately doesn’t automate — and the architecture review approves it.

Same skills, same code — only the inputs change. That is the whole point.

01Why

Every Threadlight skill except this one is opinionated and deterministic: bounded inputs, one right answer. A specific customer’s production estate is not bounded — their landing zone, identity model, regions, private network and governance are theirs, and no two customers are alike. Customizing is how you reconcile our opinions with their environment: keep what fits, override what doesn’t, and be explicit about what we deliberately don’t automate.

Our defaults are a starting position, not a mandate. A customer may be stricter than us (extra controls) or simply different (their identity model, their pipelines). Either way the skills don’t change — you change the inputs.

This is deliberately instructions, not a generator. Branching on a customer’s environment in code is the anti-pattern: it would fork the skills on every engagement and rot. The skills stay parameterised; you feed the customer’s values as inputs.

Where customization actually bites

CI/CD pipelines

Their identity model (OIDC / WIF), RBAC scoped to the spoke RG, runner topology. threadlight-cicd generates the pipeline; you feed it these constraints.

Isolated / private-VNet

Fully-private estates with no public egress. The deploy and validation loop has to run from inside the perimeter — a public pilot doesn’t prove it.

Dev machines

An SE laptop can’t reach the customer’s private endpoints. You need a dev box inside the VNet (or a sanctioned tunnel) to run the loop the way the workload will.

02How

One mechanism makes this safe and repeatable: a pinned fork plus an overlay. You never edit skill files.

  • Fork & pin. Fork threadlight-skills into the customer / engagement org and record the exact upstream commit (SHA) in overlay/upstream-pin.md.
  • Overlay, don’t edit. Every customer input — selectors, azd env, customer-overrides.yaml, CI/CD constraints — lives in an overlay/ directory beside the untouched fork.
  • Clean merge. Because the skill files stay byte-for-byte upstream, git merge upstream/main is a clean merge. If a merge ever conflicts, an override leaked out of the overlay.
engagement-repo/
├── threadlight-skills/          # pinned upstream fork — left UNMODIFIED
├── overlay/                     # customer inputs the skills already consume
│   ├── upstream-pin.md          # the SHA you forked from
│   ├── deploy/                  # landing-zone targeting, azd env, selectors
│   ├── production-ready/        # customer-overrides.yaml, thresholds
│   ├── cicd/                    # identity model, scoped RBAC, runners
│   └── safe-check/              # resource selectors + naming standard
└── docs/threadlight-customize/  # the engagement record
    ├── customer-profile.md      # intake
    ├── customization-map.md     # keep-vs-override decisions
    └── non-coverage.md          # what we deliberately don't automate

A private-VNet target needs a dev box inside the perimeter. Three patterns, one gate.

Either way the gate is the private-VNet pre-flight: private DNS resolves to private IPs, 443 reaches every private endpoint, and control-plane egress is allowed — only then deploy. 03How to

Intake is the long pole — start it during the pilot. Everything after it is mechanical. The fork and scaffold are three commands: 

# 1 · fork into the customer org and clone (gh wires origin + upstream)
> gh repo fork aiappsgbb/threadlight-skills --org <customer-org> --clone --remote
> cd threadlight-skills

# 2 · pin the exact upstream commit you forked from
> git rev-parse upstream/main   # → record this SHA in overlay/upstream-pin.md

# 3 · scaffold the overlay — customer inputs live here, never in skill files
> mkdir -p overlay/{deploy,production-ready,cicd,safe-check} docs/threadlight-customize
  1. Intake — the long pole. Fill customer-profile.md with their platform owner: documents, environment setup, requirements, mandated template / starter code. Every blank is a blocker.
  2. Fork & pin. Fork into the customer org; record the SHA in overlay/upstream-pin.md (commands above).
  3. Scaffold the overlay. One mkdir -p stands up the four skill folders plus the engagement-record directory.
  4. Map keep-vs-override. Fill customization-map.md — name the exact selector, azd env, or threshold to change per skill. Map template ↗
  5. Drop real overrides. Put the customer’s values in overlay/ — for example the customer-overrides.yaml below. Never edit a skill file.
  6. Test inside the boundary. Pre-flight green, then run the local-test / deploy loop on the in-VNet box.
  7. Name the boundary. Record what you customized and what Threadlight deliberately doesn’t automate in non-coverage.md — the architecture review signs off on this.
  8. Pull upstream later. git merge upstream/main stays clean because the overlay held the diff; then re-pin the new SHA. Full runbook ↗

One real override

# overlay/production-ready/customer-overrides.yaml
customer: <customer-name>
overrides:
  - recipe_id: SEC-105            # should-fix: KV access via RBAC, not legacy policies
    status: pass                  # customer uses an approved equivalent control
    reason: "Governed by the customer's audited PIM workflow; reviewed 2026-Q2."
  - recipe_id: NET-103            # should-fix: NSG flow logs on spoke subnets
    status: fail                  # customer is STRICTER than our default
    reason: "Customer audit policy mandates flow logs on every spoke subnet."

Overrides are status flips only (pass ↔ fail) on should-fix findings, each with a written reason for the audit trail. A must-fix finding cannot be flipped — the scorecard exits non-zero and names it. The decision lives in the overlay, never in an edited skill.
