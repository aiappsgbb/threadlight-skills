The final leg · fork & onboard

Make Threadlight the customer’s own.

The pilot proved the agent works. The next conversation is harder: “now stand this up inside our environment — our landing zones, our identity model, our RBAC, our pipelines, our governance — and test it on our tenant.” That is production onboarding, and it is priority #1.

threadlight-customize is the leg that gets you there — deliberately as instructions and fill-in runbooks, not automation. No two customers’ onboarding is the same, so we don’t pretend to generate it. We give you the frame, the intake, and the in-environment test loop, then get out of the way.

Threadlight skills .md files a coding agent (GitHub Copilot in your IDE, Copilot CLI, or the GitHub chat panel) reads and follows. Not commands. Not a CLI of our own.

Why this leg exists

A green pilot is not a production system. The gap is the customer’s environment.

The pilot proved the agent works — on an SE sandbox, with public endpoints and your subscription. “Production” means the same agent living inside their estate: their landing zones, their identity model, their RBAC, their private network, their governance. Closing that gap is the whole job — and it is different for every customer.

What the pilot proved

The agent is real

Design, deploy, evals, a green production-ready scorecard — the idea works and is production-shaped. That earns the next conversation; it does not finish it.

What it didn’t

It never met their estate

A sandbox run says nothing about private DNS, scoped identity, mandated IaC, change-management gates, or whether deploy even reaches the customer’s private endpoints. That is where onboarding lives.

Why fork, not rebuild

Make it theirs — keep upstream

Forking lets you tailor the pipeline to one customer and still pull upstream Threadlight fixes later, because every customer change lives in an overlay — not in edited skill files.

Scope, on purpose

Every other skill is a generator. This one isn’t — by design.

The rest of Threadlight is opinionated and deterministic because its inputs are bounded — a SPEC section, a set of selectors, an azd env. A customer’s production onboarding is not bounded. Trying to cover every permutation in code produces a generator that is wrong for everyone.

Bounded → automate

What the pipeline already generates

Design, deploy, safe-check, evals, the production-ready scorecard, the CI/CD pipeline — all deterministic, all tested, because the shape of the input is known. Those skills stay exactly as they are.

Unbounded → instruct

What no generator can own

Landing-zone topology, identity model, allowed regions, private DNS, egress rules, mandated IaC modules, change-management gates, approval chains — all theirs, all different. So this leg is prose + templates: capture, classify, test, and name the boundary.

It deliberately does not generate the onboarding for you. That honesty is a feature — it keeps the customer’s architecture review calibrated and stops the next engagement from assuming the work is turnkey.

Run them in order

Four moves. Each produces one durable artifact.

Every move drops a file under docs/threadlight-customize/, so the engagement is auditable and resumable. Start Move 1 early — intake is the long pole, and you can run it in parallel with the pilot.

Move 1 · the long pole

Intake gate

You cannot customize what you have not been given. Fill the customer-profile workbook with the customer’s platform owner: documents, environment setup, requirements, and mandated template / starter code. Treat every blank as a blocker, not a TODO.

→ customer-profile.md

Move 2 · fork vs keep

Customization map

Decide, per skill, what you override and what you leave alone. The map classifies every Threadlight skill as customer-agnostic or needs per-customer override, and names the exact SPEC section / selector / azd env hook to change.

→ customization-map.md

Move 3 · the stuck point

Test in the customer’s env

A pilot that only ever ran on your sandbox is not onboarded. Run the local-test / deploy loop inside the customer’s boundary — usually a fully-private VNet with no public egress. Two patterns work; a pre-flight checklist gates the deploy.

→ green private-VNet pre-flight + a run inside the env

Move 4 · close honestly

Name the boundary

Write down the seams you customized (so the next SE can read the diff) and what Threadlight deliberately does not automate for this customer, with a short decision log. The non-coverage doc is what keeps the CISO/architecture review calibrated.

→ non-coverage.md

Where the environment bites

Production onboarding is priority #1.

The customization map flags four skills as priority — they are the production-onboarding leg, and they are where the customer’s real environment first pushes back. These get your override budget first.

threadlight-deploy priority

Landing-zone targeting (sub / RG), resource selectors, region allow-list, and the private-endpoint + private-DNS wiring that turns a public pilot into something that lives behind the customer’s perimeter.

threadlight-safe-check priority

The customer’s resource selectors and naming standard that the pre/post-deploy gate validates against — so “green” means green in their tenant, not on your sandbox.

threadlight-cicd priority

Pipeline shape, federated-identity model, scoped RBAC, runner topology. This skill generates the pipeline — you feed it the customer’s constraints from intake.

threadlight-production-ready priority

Pillar thresholds and customer_overrides tuned to the customer’s policy baseline, so the scorecard speaks their standard at the architecture review.

Keep these overrides in an overlay, never by editing forked skills in place — so you can still pull upstream Threadlight fixes. That’s the fork mechanics below.

The part the field kept getting stuck on

Run the loop inside the customer’s boundary.

Fully-private (private-VNet) environments with no public egress are where onboarding actually proves out — and where an SE laptop can’t reach. Two patterns make the dev/test loop work from inside the perimeter, gated by one pre-flight.

Recommended

Azure ML compute instance + VS Code

A compute instance with no public IP, joined to the customer’s VNet, driven over VS Code Remote. Reaches the Foundry, ACR, Cosmos, and Key Vault private endpoints the way the real workload will — the closest thing to running “on their tenant.”

Quick box

GitHub Codespaces

A fast, disposable dev box for the loop. With Codespaces private networking it reaches into the customer’s estate — honestly hedged: availability depends on the customer’s GitHub plan and network posture, so the runbook says when to reach for Azure ML instead.

The gate

Private-VNet pre-flight

Before any deploy: confirm private DNS resolution, egress posture, and reachability of every private endpoint. A half-resolved DNS zone fails the deploy in a way that looks like an auth error and burns an afternoon — the checklist catches it first.

Do this once, up front

Fork the way that survives upstream updates.

The point of forking is to customize and still pull upstream Threadlight fixes later. Three steps keep both true.

Step 1

Fork

Fork threadlight-skills (or install the plugin and overlay alongside it) into the customer’s org or your engagement repo.

Step 2

Pin upstream

Record the upstream commit you forked from — mirror the upstream-pin.md convention threadlight-deploy and threadlight-local-test already use.

Step 3

Overlay, don’t fork-edit

Keep customer overrides (selectors, env hooks, thresholds, mandated IaC) in a separate overlay layered over the pinned upstream, so a later git merge of upstream doesn’t clobber the customizations.

Move 4, restated

The boundary is the deliverable.

Closing the engagement honestly is not an afterthought — it is the artifact that keeps expectations true. Name what you customized, name what Threadlight does not automate for this customer, and log the decisions. If you find yourself writing a generator that branches on the customer’s landing zone, stop: that seam is exactly what this skill keeps as instructions.

How to start

Day one. Fork, pin, and open the intake.

You don’t need the whole customization map to begin. Three commands stand up the fork and the pin; one conversation opens the intake — the long pole. Here is the literal first hour. 

# 1 · fork into the customer org and clone (gh wires origin + upstream)
> gh repo fork aiappsgbb/threadlight-skills --org <customer-org> --clone --remote
> cd threadlight-skills

# 2 · pin the exact upstream commit you forked from
> git rev-parse upstream/main   # → record this SHA in overlay/upstream-pin.md

# 3 · scaffold the overlay — customer inputs live here, never in skill files
> mkdir -p overlay/{deploy,production-ready,cicd,safe-check} docs/threadlight-customize

Then, in order

Open the intake start here

Copy customer-profile.md.tmpl and work it with the customer’s platform owner. Start it in parallel with the pilot — it is the long pole, and every blank field is a future blocked deploy.

Next

Map fork-vs-keep

Fill the customization map: per skill, keep or override. The four production-onboarding skills get your override budget first; each override drops a file into the matching overlay/ folder.

Then prove it

Test inside their boundary

Stand up the in-env loop (Azure ML VS Code or Codespaces), clear the private-VNet pre-flight, and run deploy from inside the customer’s perimeter — not your sandbox.
