Threadlight / Efficiency

Strong-model quality, at a cheap-model price

Threadlight routes each turn to the smallest model that can do the job — cheap by default, escalating only the hard turns to a stronger model. We don’t pitch that saving on a slide. We prove it in the same CI pipeline that ships the product — scored on quality and real Azure dollars, across a simple workload and a genuinely hard one. And every run, pass or fail, feeds a cold-path that turns the CI logs into the next improvement.

Routed, not fixed

A model-router keeps routine turns on gpt-5.4-mini and escalates only the hard ones to gpt-5.4. One deployment, per-turn economics — you pay the premium rate only when the work actually needs it.

Measured in CI

The real threadlight-e2e pipeline runs a controlled head-to-head: a cheap baseline, the router, and a quality ceiling — on a simple and a hard workload, scored on convergence, rounds, and billed Azure spend.

Self-improving

A cold-path (learn) reads any single run — no baseline needed — classifies what went wrong, and emits ranked, actionable findings. The pipeline gets better on its own.

The cheapest per-run price is a false economy when the run doesn’t finish — you pay again in reruns and stalled demos. Routing buys you the finish rate of the strong model while keeping the majority of turns on the cheap one.

01Why route at all

A single fixed model forces one bad trade. Pin everything to a cheap model and it flies on routine turns but stalls the moment the work gets hard — more rounds, more retries, or it simply never converges. Pin everything to a strong model and it finishes, but you pay the premium rate on every trivial turn — the file reads, the boilerplate, the “yes, continue.”

A model-router refuses the trade. It keeps the routine turns cheap and escalates only the turns that need reasoning. Done well, it captures the finish rate of the strong model while paying cheap-model rates on the majority of turns. That is the whole efficiency story — and it’s also a story worth telling a customer: we spend their compute the way we’d spend our own.

We deliberately constrain the router to {gpt-5.4, gpt-5.4-mini} and exclude the premium gpt-5.5 tier. The point is efficiency, not maximalism: show that a disciplined two-model ladder gets the job done, rather than reaching for the most expensive model on the shelf.

02How we prove it

Claims about efficiency are cheap; a run in the real pipeline is not. So we validate the router the same way we validate everything else — through the full threadlight-e2e pipeline: design → deploy (azd up) → live agent invoke, in a clean Actions runner against a real Foundry account.

The experiment is a controlled matrix: three model arms × two workloads, each run end-to-end.

Arms cheap baseline · gpt-5.4-mini model-router · {5.4, 5.4-mini} quality ceiling · gpt-5.4
Workloads returns-triage · simple fsi-kyc-aml · genuinely hard

The simple workload is a sanity floor; the hard one — a multi-persona KYC/AML agent — is where a cheap model is supposed to break down. Each cell is scored on three axes that a customer actually cares about:

  • Did it finish? — the pipeline’s own quality gates (design conforms, it deploys, the live agent answers).
  • How much work? — the number of agent rounds it took to get there.
  • What did it cost? — real billed Azure spend, read back from the Cognitive Services account after the run.

We run the matrix repeatedly and fix what it surfaces between runs — three matrices so far. Numbers below are directional (one run per cell); the signal is in the shape, not the third decimal place.

03What we found

Two clean signals, both intuitive once you see them. On the simple workload the cheap model keeps up — routing’s overhead isn’t worth it there. On the hard workload the cheap model falls behind: it can’t recover from a mid-pipeline blocker and times out, while the router tracks the strong model’s trajectory and improvises past the blocker — at a fraction of the ceiling’s cost.

The self-improving part

The most durable win isn’t a single number — it’s that the pipeline improves itself. A cold-path called learn reads any one run (no paired baseline required), strips the log noise, classifies the anomalies — missing skill dependency, protocol-contract mismatch, rate-limit wall — and emits a ranked list of concrete fixes.

In this very exercise it earned its keep: from the failed CI logs it surfaced a missing skill dependency and a protocol-contract gap that were quietly stalling the hard workload. We shipped three fixes straight from those logs — no guesswork, no re-running pairs to find them. That is the loop we want a customer to see: not “it worked once,” but “it tells us how to make it work better every time.”

  1. Pin the router. Constrain the model-router deployment to {gpt-5.4, gpt-5.4-mini} and record the pool so it restores cleanly.
  2. Run the matrix. Dispatch all three arms across both workloads through threadlight-e2e — same pipeline, same runner, only the model wiring changes.
  3. Score on quality + cost. Ingest each run, read billed Azure spend back from the account, and rank by convergence and dollars.
  4. Mine every run. Point learn at each run to turn the logs into ranked, actionable findings — pass or fail.
  5. Fix and re-run. Apply the fixes, run the next matrix, and watch the verdict tighten. Efficiency you can defend, not just assert.
