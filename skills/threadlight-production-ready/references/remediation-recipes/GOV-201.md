## Target file
N/A — this is a subscription / resource-group governance gate. No repo-local file to edit. The Azure Policy assignments live in the tenant, not in the code repo.

## Edit type
`manual` — requires central team involvement or Owner-level subscription permissions.

## Edit recipe
Azure Policy assignments must be **present at subscription scope or resource-group scope** for the target environment. v0.4.0 threadlight checks for the presence of **any** Azure Policy assignments; future versions will check for specific policy IDs (e.g., "Deny public storage", "Require private endpoint").

**If the central security/governance team owns policy assignments (most common case):**

1. File a ticket with your central security or governance team with this template:

```
Subject: [threadlight-production-ready v0.4.0] Request Azure Policy assignment for <pilot-name>

Required policies for production readiness:
- At minimum, ensure at least one Azure Policy assignment exists at subscription or RG scope.
- Recommended baseline (ASB v3 or equivalent):
  - Deny public storage accounts without private endpoint
  - Require private link for key dependencies (Key Vault, Cosmos DB, Container Registry)
  - Audit VNet service endpoints vs private endpoints
  - Require encryption at transit and rest

Target subscription: <SUB_ID>
Target resource group: <RG_NAME>

Pilot: <pilot-name>
SRE point of contact: <email>
```

2. Wait for the team to confirm assignment completion. They will use:
   ```bash
   az policy assignment create \
     --scope "/subscriptions/<SUB>" \
     --policy "<POLICY_ID_OR_NAME>" \
     --display-name "Required production policy"
   ```

**If your team has Owner role and can apply policies directly (self-service):**

1. List available built-in policies:
   ```bash
   az policy definition list --query "[].{id:id, name:name}" -o table | head -20
   ```

2. Assign the Azure Security Benchmark (ASB v3) initiative (strongest baseline):
   ```bash
   az policy assignment create \
     --scope "/subscriptions/<SUB>" \
     --policy-set-definition "/providers/Microsoft.Authorization/policySetDefinitions/1f3afdf9-d0c9-4c3d-847f-89da613e70a8" \
     --display-name "Azure Security Benchmark v3 - ASB" \
     --assignment-scope "/subscriptions/<SUB>"
   ```

3. Or assign specific policies (e.g., deny public storage):
   ```bash
   az policy assignment create \
     --scope "/subscriptions/<SUB>/resourceGroups/<RG>" \
     --policy "Deny public blob storage accounts" \
     --display-name "No public blobs"
   ```

4. Verify assignment:
   ```bash
   az policy assignment list \
     --scope "/subscriptions/<SUB>/resourceGroups/<RG>" \
     --query "[].{name:name, policyDefinitionId:policyDefinitionId}" -o table
   ```

## Verification
Re-run threadlight: `python3 scripts/production_ready.py --target-rg <RG> --target-sub <SUB>`. GOV-201 should flip from `fail` to `pass` once at least one policy assignment exists.

Expected: `az policy assignment list -g <RG>` returns at least one assignment.

**Note:** Non-compliance under an assignment (GOV-202) is a separate check. This recipe only ensures the assignment itself exists. After assignment, monitor compliance:
```bash
az policy state list --resource-group <RG> --filter "isCompliant eq false" | head -10
```
