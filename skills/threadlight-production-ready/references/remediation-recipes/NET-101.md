## Target file
`infra/foundry/account.bicep` (or the file containing the `Microsoft.CognitiveServices/accounts` resource for Foundry; see NET-003 for alternatives).

## Edit type
`replace` — same as NET-003; this is the live check that confirms the Bicep change has been deployed to Azure

## Edit recipe

1. This finding is a **live check** that runs after deployment. It verifies that the actual Foundry account in Azure has `publicNetworkAccess=Disabled`.
2. The fix is already described in **NET-003**: ensure the Bicep includes `publicNetworkAccess: 'Disabled'` in the Foundry resource properties.
3. After editing `infra/foundry/account.bicep` per NET-003, redeploy the infrastructure:

```bash
azd deploy
# or if using manual Bicep deployment:
az deployment group create \
  --resource-group <RG> \
  --template-file infra/main.bicep \
  --parameters infra/main.bicepparam
```

4. Wait for the deployment to complete successfully. The Foundry account's `publicNetworkAccess` property will update to `Disabled` during the deployment.

## Verification

Once the deployment is complete, re-run threadlight to verify the live state:

```bash
python3 scripts/production_ready.py --target-rg <RG> --target-sub <SUB>
```

NET-101 should flip from `must-fix` to `pass`. The live check queries `az cognitiveservices account list -g <RG>` and confirms all accounts have `publicNetworkAccess=Disabled`.

If the check still fails after deployment:
- Manually verify the account state: `az cognitiveservices account show -n <foundry-name> -g <RG> --query "properties.publicNetworkAccess"`
- Expected output: `"Disabled"`.
- If the output shows `"Enabled"` or is missing, the Bicep change was not applied; re-check the resource definition and redeploy.
