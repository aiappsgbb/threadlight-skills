## Edit recipe
1. Create the directory `docs/runbooks/` if it does not exist.
2. Create a new file `docs/runbooks/incident-response.md` with the following content:

```markdown
# Incident Response Runbook

## Overview
This runbook guides the on-call engineer through common production incidents in [pilot name].

## Severity Levels

- **Sev 1:** Service down or critical functionality unavailable. Initiate war room. Target: 15 min response.
- **Sev 2:** Degraded service, limited impact. Target: 30 min response.
- **Sev 3:** Non-critical issue or alert. Target: 4h response.

## Alert Routing

All alerts route to: [Action Group Name / Teams channel]
Escalation: L1 → L2 → L3 (see SPEC § 12)

## Common Playbooks

### Playbook 1: High Latency / Timeouts

1. Check [resource] metrics in Azure Monitor.
2. Review application logs in Log Analytics.
3. If CPU > 80%, scale up or review code.
4. If timeout errors, check downstream dependencies (database, API).
5. Page L2 if not resolved in 10 minutes.

### Playbook 2: Failed Deployments

1. Check deployment status: `az deployment group show -g <RG> -n <deployment-name>`
2. Review error logs in the deployment output.
3. Rollback if necessary: `az deployment group cancel -g <RG> -n <deployment-name>`
4. Engage the product team and page L2.

### Playbook 3: Quota / Capacity Exhausted

1. Check quotas in Azure Portal or via `az quota list`.
2. If Foundry PTU is exhausted, check priority of pending requests in Azure OpenAI Studio.
3. Request quota increase from central team or enable PAYG fallback.
4. Page L2 or finance to discuss capacity plan.

## Escalation

1. **L1 (On-call):** Triage and execute playbooks. Escalate to L2 if unresolved in 15 minutes (Sev 1) or 30 min (Sev 2).
2. **L2 (Backup):** Code review, deeper troubleshooting, customer communication.
3. **L3 (Lead):** Architecture decisions, policy exceptions, post-mortem scheduling.

## Postmortem

After any Sev 1 or Sev 2 incident, schedule a postmortem within 24 hours. Document in `docs/postmortems/`.

---

**Last updated:** [ISO date]  
**Reviewed by:** [SRE team]
```

3. Adjust the content to match your specific application (replace placeholders like `[pilot name]`, `[resource]`, etc.).
4. If multiple runbooks exist (e.g., for different components), create them in `docs/runbooks/` and link them from a central `docs/runbook.md` index.
