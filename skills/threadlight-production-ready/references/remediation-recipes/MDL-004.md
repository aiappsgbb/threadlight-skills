## Target file
`infra/foundry/deployments.bicep` (or wherever the `Microsoft.CognitiveServices/accounts/deployments` resources are declared).

## Edit type
`insert`

## Edit recipe
1. Locate all `Microsoft.CognitiveServices/accounts/deployments@<version>` resources in the Bicep.
2. For each deployment, ensure the `sku` block includes a `capacity` field:

   ```bicep
   resource gpt 'Microsoft.CognitiveServices/accounts/deployments@2024-10-01' = {
     parent: foundry
     name: 'gpt-4o-2024-08-06'
     sku: {
       name: 'Standard'
       capacity: 10  // ← ADD THIS if missing
     }
     properties: {
       model: {
         format: 'OpenAI'
         name: 'gpt-4o'
         version: '2024-08-06'
       }
       raiPolicyName: 'Microsoft.DefaultV2'
     }
   }
   ```

3. If `sku.capacity` already exists, verify it reflects your planned TPM (tokens per minute) for production. Use the `paygo-ptu-cost-analyzer` skill to calculate the capacity needed based on your expected token throughput.
4. Typical capacity values: `10` (PAYG standard), `20–100` (scaled PAYG), `1–64` (PTU reservations). Align with your SPEC § 12 declarations.

## Verification
1. Re-run threadlight: `python3 scripts/production_ready.py --target-rg <RG> --target-sub <SUB>`.
2. MDL-004 should flip from `fail` to `pass` (capacity is declared).
3. Verify the Bicep compiles: `az bicep build --file infra/foundry/deployments.bicep`.
4. Cross-check capacity against SPEC § 12: the declared TPM in the design doc should match the capacity × throughput model of the SKU.
