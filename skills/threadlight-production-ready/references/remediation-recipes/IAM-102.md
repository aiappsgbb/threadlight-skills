## Target file
`infra/identity.bicep` (or wherever `Microsoft.Authorization/roleAssignments` grant Owner or Contributor to workload identities). The live check flags any Owner or Contributor assignments on the RG or subscription as must-fix.

## Edit type
`manual` — DO NOT auto-revoke. Removing Owner/Contributor from a workload identity can break running workloads if done without verification.

## Edit recipe
Present this to the operator before making any change:

> The threadlight assessment detected that the workload identity `<identity-name>` has been granted **Owner** or **Contributor** role at the resource group or subscription scope. This is a security violation — workload identities should have least-privilege roles (e.g., `KV Secrets User`, `Storage Blob Data Reader`, `Cosmos Data Contributor`).
>
> **Implications of revoking:**
> - The workload may temporarily lose access to resources it depends on (e.g., databases, storage accounts).
> - You must replace the broad roles with specific least-privilege roles BEFORE revoking, or the workload will fail.
>
> **If you proceed:**
> 1. The agent will audit what the identity currently accesses (if possible).
> 2. Update `infra/identity.bicep` to remove Owner/Contributor and add least-privilege roles instead.
> 3. Redeploy Bicep.
> 4. Test the workload to confirm it still functions.
>
> **If you decline:** Mark this finding as `accepted-risk` in `waivers.yaml` with a signed justification.

The operator must explicitly confirm before the agent proceeds. If confirmed:

1. Audit the workload's actual access needs:

   ```bash
   # Check what roles would be required (manual review):
   # - Does it read from Cosmos DB? → Cosmos Data Reader
   # - Does it write to Storage? → Storage Blob Data Contributor
   # - Does it read secrets from KV? → KV Secrets User
   # etc.
   ```

2. Update `infra/identity.bicep` to replace broad roles with least-privilege:

   ```bicep
   // Remove:
   // resource ownerRole 'Microsoft.Authorization/roleAssignments@2022-04-01' = { ... Owner ... }

   // Replace with:
   resource kvSecretsUser 'Microsoft.Authorization/roleAssignments@2022-04-01' = {
     name: guid(resourceGroup().id, 'KV Secrets User', workloadIdentityPrincipalId)
     scope: resourceGroup()
     properties: {
       principalId: workloadIdentityPrincipalId
       roleDefinitionId: '/subscriptions/${subscription().subscriptionId}/providers/Microsoft.Authorization/roleDefinitions/4633458b-17de-408a-b874-0445c86b69e6'  // KV Secrets User
     }
   }

   // Add other needed roles similarly...
   ```

3. Redeploy:

   ```bash
   azd up
   ```

4. Test the workload to confirm functionality.

If the operator declines, file a waiver entry:

```yaml
- finding_id: IAM-102
  status: accepted-risk
  signed_by: <operator-email>
  justification: "Workload identity requires Owner role for legacy authorization patterns; migration planned for Q3 2024"
  expires: <ISO-8601 date>
```

## Verification
Re-run threadlight: `python3 scripts/production_ready.py --target-rg <RG> --target-sub <SUB>`. IAM-102 should flip from `fail` to `pass` (no Owner/Contributor assignments detected on RG).

Confirm:

```bash
az role assignment list --resource-group <RG> --subscription <SUB> \
  --query "[?roleDefinitionName=='Owner' || roleDefinitionName=='Contributor']"
```

Should return empty list.
