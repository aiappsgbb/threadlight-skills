## Target file
Every MCP server config the scan flagged: `.mcp.json`, `mcp.json`, `.vscode/mcp.json`, `.cursor/mcp.json`, or the `mcpServers` block of any agent config — specifically the `env` values that carry a live secret.

## Edit type
`replace`

## Edit recipe
A committed secret (an API key, token, or connection string hard-coded in an `env` block) is exposed to everyone with repo access and survives in git history. Replace each inline secret with an indirection that resolves at runtime:

1. **Prefer environment indirection** — reference a variable the runtime injects, never the literal value:
   - **Before:**
     ```json
     "env": { "API_KEY": "sk-live-0123456789abcdefabcdef" }
     ```
   - **After:**
     ```json
     "env": { "API_KEY": "${env:MCP_EXAMPLE_API_KEY}" }
     ```

2. **Source the secret from Azure Key Vault**, so the platform holds it and issues short-lived access — the repo never sees the value:
   ```bash
   az keyvault secret set \
     --vault-name <your-keyvault> \
     --name mcp-example-api-key --value <secret>
   ```
   Inject it into the process environment at deploy/run time (Key Vault reference in your Container App / App Service / Functions settings, or a `secretRef` in your pipeline). If you consume MCP through **foundry-toolbox**, bind the credential through the toolbox's Key Vault integration instead of inlining it — the platform brokers the secret for you.

3. **Rotate any secret that was ever committed.** Removing it from the file is not enough; it remains in history. Issue a new secret in Key Vault, update the reference, and revoke the exposed one.

4. Re-run the producer to confirm no server still carries inline credentials:
   ```bash
   python3 scripts/mcp_sbom.py --root . --out mcp-sbom.json
   ```
