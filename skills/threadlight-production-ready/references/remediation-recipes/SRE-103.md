## Target file
`docs/sre/runbook.md`.

## Edit type
`auto-apply` — a remediation agent may create the runbook if it is missing. The production-readiness assessor itself must only read the repo and report whether the file exists.

## Edit recipe
If `docs/sre/runbook.md` is missing, create the `docs/sre/` directory and write this starter runbook. Replace every placeholder with customer-specific values before committing:

```markdown
# SRE Runbook

## Service overview

- Service name: <name>
- Production environment: <subscription/resource group/region>
- Primary owner: <team or person>
- Support hours: <coverage window>

## Health checks

- Public endpoint or synthetic check: <URL or monitor>
- Azure resource health checks: <links or commands>
- Key dashboards: <workbook, Grafana, or portal links>

## Incident response

1. Confirm customer impact and severity.
2. Check current alerts, recent deployments, and dependency health.
3. Page the on-call owner through <paging mechanism>.
4. Start an incident timeline and record mitigation decisions.

## Escalation

- Primary on-call: <rotation link>
- Engineering escalation: <team or channel>
- Customer/comms escalation: <team or channel>

## Recovery procedures

- Rollback procedure: <link or steps>
- Backup/restore runbook: <link or steps>
- Known failure modes: <link or notes>
```

If the customer already has an SRE runbook elsewhere, add this file as a discoverability shim that links to the authoritative runbook, or file a waiver explaining the exception.
