## Target file
`infra/foundry/kv.bicep` (or create this file if it does not exist; if KV is declared in a different Bicep file, search the repo for `Microsoft.KeyVault/vaults` and edit that file instead).

## Edit type
`insert` — add a new Key Vault resource to the Bicep deployment if none exists, or consolidate into a single KV module if multiple exist.

## Edit recipe
1. Create or open `infra/foundry/kv.bicep`.
2. Add the following Key Vault resource declaration (or update the existing one):

```bicep
param location string = resourceGroup().location
param kvName string
param tenantId string = subscription().tenantId

resource keyVault 'Microsoft.KeyVault/vaults@2023-07-01' = {
  name: kvName
  location: location
  properties: {
    tenantId: tenantId
    sku: {
      family: 'A'
      name: 'standard'
    }
    enableRbacAuthorization: true
    enableSoftDelete: true
    enablePurgeProtection: true
    softDeleteRetentionInDays: 90
    publicNetworkAccess: 'Disabled'
    networkAcls: {
      bypass: 'AzureServices'
      defaultAction: 'Deny'
    }
  }
}

output kvId string = keyVault.id
output kvName string = keyVault.name
```

3. Ensure `infra/main.bicep` or the top-level orchestration file includes this KV module:

```bicep
module keyVault './foundry/kv.bicep' = {
  name: 'kv-deployment'
  params: {
    kvName: 'kv-${uniqueString(resourceGroup().id)}'
    location: location
  }
}
```

4. Save the file and commit it to the repo.

## Verification
1. Run Bicep compilation: `az bicep build --file infra/main.bicep`
2. Inspect the compiled ARM template (JSON) to verify `Microsoft.KeyVault/vaults` resource appears with `enableSoftDelete: true` and `enablePurgeProtection: true`.
3. Re-run threadlight: `python3 scripts/production_ready.py --target-rg <RG> --target-sub <SUB>`. SEC-001 should flip from `must-fix` to `pass`.
