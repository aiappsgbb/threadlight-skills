## Target file
`infra/foundry/kv.bicep` (or wherever the `Microsoft.KeyVault/vaults` resource is declared — search the repo for `Microsoft.KeyVault/vaults` if the canonical path differs).

## Edit type
`replace` — update the existing Key Vault resource to disable public network access and enforce private endpoint connectivity.

## Edit recipe
**Prerequisite:** Ensure a private endpoint to Key Vault is already declared in `infra/foundry/private-endpoint.bicep` (see NET-002 remediation recipe). Public network access cannot be safely disabled without a private endpoint to access the vault.

1. Open `infra/foundry/kv.bicep` in your editor.
2. Locate the `Microsoft.KeyVault/vaults` resource declaration.
3. Inside the `properties` block, add or update the network configuration:

   ```bicep
   resource keyVault 'Microsoft.KeyVault/vaults@2023-07-01' = {
     name: kvName
     location: location
     properties: {
       tenantId: tenantId
       sku: {
         family: 'A'
         name: 'standard'
       }
       enableRbacAuthorization: true
       enableSoftDelete: true
       enablePurgeProtection: true
       softDeleteRetentionInDays: 90
       publicNetworkAccess: 'Disabled'   // <-- Set to 'Disabled'
       networkAcls: {
         bypass: 'AzureServices'        // <-- Allow Azure services (Functions, etc.)
         defaultAction: 'Deny'          // <-- Deny by default
       }
     }
   }
   ```

4. Ensure the private endpoint is configured to use `privateLinkServiceConnections` with `status: 'Approved'` in `private-endpoint.bicep`:

   ```bicep
   resource kvPrivateEndpoint 'Microsoft.Network/privateEndpoints@2023-06-01' = {
     name: 'pe-kv'
     location: location
     properties: {
       subnet: {
         id: subnetId
       }
       privateLinkServiceConnections: [
         {
           name: 'pe-kv-connection'
           properties: {
             privateLinkServiceId: keyVault.id
             groupIds: [ 'vault' ]
             privateLinkServiceConnectionState: {
               status: 'Approved'
               description: 'Auto-approved for production posture'
             }
           }
         }
       ]
     }
   }
   ```

5. Save the file and commit:
   ```bash
   git add infra/foundry/kv.bicep infra/foundry/private-endpoint.bicep
   git commit -m "Disable public network access on Key Vault (SEC-103)

   - Set publicNetworkAccess: Disabled
   - Configured networkAcls.defaultAction: Deny
   - Private endpoint connection approved
   - Cross-ref: NET-002 (private endpoint established)
   
   Fixes SEC-103"
   ```

6. Deploy the Bicep changes to Azure:
   ```bash
   az deployment group create \
     --resource-group <RG> \
     --template-file infra/main.bicep
   ```

## Verification
1. After deployment, verify public network access is disabled:
   ```bash
   az keyvault show -n <kvname> -g <RG> --query "properties.publicNetworkAccess"
   ```
   Expected output: `"Disabled"`

2. Confirm network ACLs are configured:
   ```bash
   az keyvault show -n <kvname> -g <RG> --query "properties.networkAcls"
   ```
   Expected output:
   ```json
   {
     "bypass": "AzureServices",
     "defaultAction": "Deny"
   }
   ```

3. Verify the private endpoint is connected and approved:
   ```bash
   az network private-endpoint show -n pe-kv -g <RG> --query "privateLinkServiceConnections[0].privateLinkServiceConnectionState"
   ```
   Expected output:
   ```json
   {
     "status": "Approved",
     "description": "Auto-approved for production posture"
   }
   ```

4. Test access from within the VNet (e.g., Function App with private endpoint):
   ```bash
   curl https://<kvname>.vault.azure.net/secrets/?api-version=2016-10-01 \
     -H "Authorization: Bearer $(az account get-access-token --query accessToken -o tsv)"
   ```
   Should succeed (HTTP 200).

5. Test from outside the VNet (public internet):
   ```bash
   curl https://<kvname>.vault.azure.net/secrets/?api-version=2016-10-01
   ```
   Should fail (HTTP 403 Forbidden or timeout).

6. Re-run threadlight: `python3 scripts/production_ready.py --target-rg <RG> --target-sub <SUB>`. SEC-103 should flip from `must-fix` to `pass`.

If access from within the VNet fails:
- Confirm the application/VM has a managed identity assigned and `Key Vault Secrets User` RBAC role.
- Verify the private endpoint's DNS records are resolving to the private IP:
  ```bash
  nslookup <kvname>.vault.azure.net
  ```
- Check that private DNS zone is linked to the VNet (if using).

If still failing after all checks, NET-002 may not be fully remediated — verify the private endpoint connection is approved and the VNet peering (if multi-spoke) is correctly configured.
