summary: Remove secrets from .azure env files and move to Key Vault
target_file: .azure/*.env or .env.*.local
edit_type: delete
---

## Target file
`.azure/**/.env`, `.env.*.local`, or any environment file committed to the repo that contains secret-like values (connection strings, API keys, tokens, passwords).

Typical locations:
- `.azure/.env`
- `.azure/.env.prod`
- `.env.local`
- `config/.env.prod`

## Edit type
`delete` — remove secret values from committed env files and replace with Key Vault references or placeholders.

## Edit recipe
1. **Identify env files with secrets:**
   ```bash
   find .azure -name ".env*" -type f
   grep -r "password\|token\|key\|connection" .azure/ --include="*.env*"
   ```

2. **For each file containing secrets, remove the sensitive values:**
   
   Before:
   ```
   AZURE_STORAGE_CONNECTION_STRING=DefaultEndpointsProtocol=https;AccountName=myaccount;AccountKey=...
   AZURE_COSMOS_KEY=abcd1234efgh5678...
   API_KEY=sk_live_1234567890
   ```

   After:
   ```
   # Secrets moved to Key Vault — see infra/foundry/kv.bicep
   # AZURE_STORAGE_CONNECTION_STRING=@Microsoft.KeyVault(SecretUri=https://...vault.azure.net/secrets/storage-connection/)
   # AZURE_COSMOS_KEY=@Microsoft.KeyVault(SecretUri=https://...vault.azure.net/secrets/cosmos-key/)
   # API_KEY=@Microsoft.KeyVault(SecretUri=https://...vault.azure.net/secrets/api-key/)
   ```

   Or replace with non-secret placeholders:
   ```
   # Note: Secrets stored in Key Vault, not in version control
   AZURE_STORAGE_CONNECTION_STRING=<move-to-kv>
   AZURE_COSMOS_KEY=<move-to-kv>
   API_KEY=<move-to-kv>
   ```

3. **Ensure all secrets are stored in Key Vault:**
   ```bash
   az keyvault secret set --vault-name <kvname> --name storage-connection --value "<connection-string>"
   az keyvault secret set --vault-name <kvname> --name cosmos-key --value "<key>"
   az keyvault secret set --vault-name <kvname> --name api-key --value "<key>"
   ```

4. **Update `.gitignore` to prevent re-commits:**
   ```
   .azure/.env
   .azure/.env.*.local
   .env.local
   .env.*.local
   ```

5. **For azd workflows, ensure secrets are injected at deploy time:**
   - Add a post-provision hook in `.azure/config.json` (if using azd):
     ```json
     {
       "hooks": {
         "postprovision": ["scripts/inject-kv-secrets.sh"]
       }
     }
     ```
   - Or manually inject secrets before deployment:
     ```bash
     export AZURE_STORAGE_CONNECTION_STRING=$(az keyvault secret show --vault-name <kvname> --name storage-connection --query value -o tsv)
     azd deploy
     ```

6. **Commit the cleanup:**
   ```bash
   git add .azure/ .gitignore
   git commit -m "Remove secrets from .azure env files; migrate to Key Vault

   - Removed hardcoded connection strings and API keys from .azure env
   - Secrets now stored in Key Vault (see infra/foundry/kv.bicep)
   - Updated .gitignore to prevent re-commits
   
   Fixes SEC-007"
   ```

## Verification
1. Run threadlight: `python3 scripts/production_ready.py --target-rg <RG> --target-sub <SUB>`. SEC-007 should flip from `must-fix` to `pass` (no secrets in committed .azure env files).
2. Verify Key Vault secrets are accessible:
   ```bash
   az keyvault secret list --vault-name <kvname> --query "[*].name"
   ```
3. Test deployment pipeline to ensure secrets are injected correctly at runtime:
   ```bash
   azd provision
   azd deploy
   ```
   Verify the deployed app/function can access secrets via Key Vault references.

If threadlight still fails:
- Check for uncommitted changes: `git status`
- Search for remaining secrets: `grep -r "password\|token\|key" .azure/`
- If secrets remain in git history, use `git filter-branch` or similar to purge them.
