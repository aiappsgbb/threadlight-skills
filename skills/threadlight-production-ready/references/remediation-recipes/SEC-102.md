## Target file
`infra/foundry/kv.bicep` (or wherever the `Microsoft.KeyVault/vaults` resource is declared — search the repo for `Microsoft.KeyVault/vaults` if the canonical path differs).

## Edit type
`manual` — DO NOT auto-apply. Purge protection cannot be disabled once enabled; the operator must acknowledge the lock-in before the agent edits Bicep.

## Edit recipe
Present this to the operator before making any edit:

> Key Vault `<KV name>` is missing `enablePurgeProtection: true`. Enabling this is **irreversible** — once set, the vault and its secrets cannot be hard-deleted for the soft-delete retention period (default 90 days), even by subscription owners with full permissions. This lock-in is required production posture for v0.4.0.
>
> If you proceed, the agent will edit `infra/foundry/kv.bicep` to add:
>
> ```bicep
> properties: {
>   enableSoftDelete: true
>   enablePurgeProtection: true
>   softDeleteRetentionInDays: 90
> }
> ```
>
> and redeploy to Azure. After redeployment, the vault is locked for 90 days and cannot be purged even by the subscription owner.
>
> If you decline, mark this finding as `accepted-risk` in the waivers file (`waivers.yaml`) with a signed justification, and continue.

The agent must wait for an explicit yes/no before editing. If the operator declines, write a waiver entry rather than skipping silently:

```yaml
- finding_id: SEC-102
  status: accepted-risk
  signed_by: <operator email>
  justification: <reason — e.g. "pilot environment, will enable before GA">
  expires: <ISO-8601 date>
```

If the operator confirms:

1. Edit `infra/foundry/kv.bicep` and ensure `enablePurgeProtection: true` is set in the Key Vault properties:

   ```bicep
   resource keyVault 'Microsoft.KeyVault/vaults@2023-07-01' = {
     name: kvName
     location: location
     properties: {
       tenantId: tenantId
       sku: {
         family: 'A'
         name: 'standard'
       }
       enableRbacAuthorization: true
       enableSoftDelete: true
       enablePurgeProtection: true    // <-- Irreversible; operator acknowledged
       softDeleteRetentionInDays: 90
       publicNetworkAccess: 'Disabled'
       networkAcls: {
         bypass: 'AzureServices'
         defaultAction: 'Deny'
       }
     }
   }
   ```

2. Commit the change:
   ```bash
   git add infra/foundry/kv.bicep
   git commit -m "Enable purge protection on Key Vault (SEC-102)

   - Set enablePurgeProtection: true (IRREVERSIBLE)
   - Vault cannot be hard-deleted for 90 days even by owner
   - Operator acknowledged this lock-in
   
   Fixes SEC-102"
   ```

3. Deploy the change:
   ```bash
   az deployment group create \
     --resource-group <RG> \
     --template-file infra/main.bicep
   ```

## Verification
Re-run threadlight after Bicep deploy. SEC-102 flips to `pass`. Confirm at the data-plane:

```bash
az keyvault show -n <kvname> -g <RG> --query "properties.enablePurgeProtection"
```

Expected output: `true`.

Document the lock-in in your change management / incident response runbooks — explicitly note that the vault cannot be emergency-deleted if compromised during the 90-day retention window. This is by design: the retention period gives time to recover from accidental deletion or detect tampering.

If the operator chose accepted-risk, threadlight should show SEC-102 as `warn` (with the waiver attribution) rather than `fail`.
