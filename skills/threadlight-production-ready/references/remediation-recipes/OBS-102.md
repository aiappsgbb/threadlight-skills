## Target file
`.github/workflows/azd-deploy-prod.yml` or equivalent CI/CD post-deploy job (created by Phase 3 CI/CD scaffold, Module D).

## Edit type
`deferred-to-pipeline` — the assessor cannot poll traces at apply time. The check must live in the production CI/CD pipeline as a post-deploy gate that verifies real traces after workload startup.

## Edit recipe
When Phase 3 (Module D) writes the CI/CD scaffold, ensure the post-deploy job contains a step that:

1. Waits for workload readiness (~30 seconds after container start).
2. Sends a test request to the agent endpoint:
   ```bash
   curl -s -X POST "https://${AGENT_ENDPOINT}/api/completions" \
     -H "Content-Type: application/json" \
     -d '{"query":"test","max_tokens":10}'
   ```
3. Waits an additional 15 seconds for trace ingestion.
4. Queries Log Analytics for ingested traces:
   ```bash
   az monitor log-analytics query \
     --workspace <workspace-id> \
     --analytics-query "traces | where timestamp > ago(24h) | take 1"
   ```
5. Asserts result row count > 0. Fail the workflow if row count = 0.

The scaffold template `references/cicd-templates/azd-deploy-prod.yml.tmpl` already contains the `smoke-otel` job stub — no additional edit is required here beyond making sure Phase 3 ran.

## Verification
Pipeline run shows `smoke-otel` job green ✓. Threadlight OBS-102 flips to `pass` once the manifest records a successful pipeline run within the last 24 hours with trace freshness confirmed.

If still failing after the smoke test passes:
1. Check the Log Analytics workspace ID is correct: `az monitor log-analytics workspace list -g <RG>`.
2. Confirm the App Insights connection is wired at account level (OBS-101).
3. Verify `configure_azure_monitor()` was called (OBS-003).

If the customer's pilot is not receiving traffic, waive OBS-102 with justification "cold start — no user traffic yet" and schedule a re-check after the agent sees real usage.
