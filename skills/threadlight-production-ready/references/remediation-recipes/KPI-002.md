## Target file

`infra/observability/alerts.bicep` (create if missing; if your alert rules live
elsewhere, search for `Microsoft.Insights/metricAlerts` or
`Microsoft.Insights/scheduledQueryRules` and add there).

## Edit type

`insert` — add a metric (or log) alert that fires when an outcome KPI drifts
away from its declared baseline. This is the deviation-alert half of the CAF
agent-observability baseline: you declare the baseline (KPI-001) **and** alert on
deviation from it (KPI-002).

## Edit recipe

1. Make sure the KPI baselines are declared first (KPI-001): target p95 latency,
   target cost-per-interaction, and target success/pass rate belong in
   `docs/SPEC.md` § 9/§ 11 so the alert thresholds below have a source of truth.
2. Create or open `infra/observability/alerts.bicep` and add a latency-deviation
   alert (repeat the pattern for a success-rate / failed-request alert):

```bicep
param appInsightsResourceId string
param appInsightsName string
param actionGroupResourceId string

// Deviation alert: server response time (latency) above the declared baseline.
// Threshold below MUST match the target p95 latency declared in SPEC § 9/§ 11.
resource latencyDeviationAlert 'Microsoft.Insights/metricAlerts@2018-03-01' = {
  name: '${appInsightsName}-latency-deviation'
  location: 'global'
  properties: {
    description: 'Agent request latency drifted above the declared KPI baseline'
    severity: 2
    enabled: true
    scopes: [appInsightsResourceId]
    evaluationFrequency: 'PT5M'
    windowSize: 'PT15M'
    criteria: {
      'odata.type': 'Microsoft.Azure.Monitor.SingleResourceMultipleMetricCriteria'
      allOf: [
        {
          name: 'LatencyP95'
          metricNamespace: 'microsoft.insights/components'
          metricName: 'requests/duration'
          operator: 'GreaterThan'
          // threshold in ms — replace 2500 with your SPEC § 9 baseline target
          threshold: 2500
          timeAggregation: 'Average'
        }
      ]
    }
    actions: [
      {
        actionGroupId: actionGroupResourceId
      }
    ]
  }
}
```

3. For a cost-per-interaction deviation, wire a `consumption budget` alert (or a
   scheduled-query alert over your cost-export table) — `threadlight-consumption-iq`
   emits the per-interaction baseline into `specs/cost-manifest.json`; alert when
   the live value exceeds it.
4. Reference the alert module from your main deploy template and pass the action
   group that notifies the on-call channel (reuse the one wired for OBS-104/105).

## Verification

- `az bicep build --file infra/observability/alerts.bicep --stdout` compiles.
- Re-run `threadlight-production-ready`; `KPI-002` flips to `pass` once an Insights
  alert referencing a KPI metric (latency / duration / cost / success / failed
  requests) is declared, and `KPI-001` is `pass` once all three baselines are
  declared in SPEC.
- Confirm the alert is bound to an action group with a real notification channel
  so a deviation actually pages someone.
