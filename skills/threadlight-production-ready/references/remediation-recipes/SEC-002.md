## Target file
Multiple files detected by SECRET_REGEX pattern matching. Typically:
- `appsettings*.json` or `local.settings.json` with embedded connection strings/passwords
- `.env` or `.env.local` files with API keys, tokens, or credentials
- Source code comments or hardcoded configuration values
- CI/CD pipeline variables or GitHub Actions secrets stored as plain text

Run `grep -r "password\|secret\|token\|key" <repo>` to locate suspect files manually.

## Edit type
`delete` — remove the hardcoded secret values and replace with Key Vault references or placeholder comments.

## Edit recipe
**CRITICAL: Rotation required before cleanup.** Hardcoded secrets are already compromised by virtue of being in version control (even in private repos, even in deleted commits). Follow this process:

1. **Before deleting anything:**
   - Identify all hardcoded secrets (search for patterns: `password=`, `ConnectionString=`, `Bearer `, `api-key:`, PEM blocks, AWS key IDs, Azure connection strings with keys).
   - Document each secret's scope: database password, service principal key, PAT, API key, etc.
   - For each secret, immediately rotate/revoke it in the live system (Azure Key Vault, GitHub, cloud provider console). This is non-negotiable—deletion from git does not invalidate leaked credentials.

2. **Remove from repo:**
   - Delete the literal values from all files (appsettings.json, .env, source code, docs, etc.).
   - Replace with placeholder comments or Key Vault reference syntax:

     ```json
     {
       "ConnectionString": "@Microsoft.KeyVault(SecretUri=https://<kvname>.vault.azure.net/secrets/<secret-name>/)"
     }
     ```

   - For source code, use environment variables:
     ```csharp
     var secret = Environment.GetEnvironmentVariable("DB_PASSWORD") 
                    ?? throw new InvalidOperationException("DB_PASSWORD not set");
     ```

3. **Update .gitignore to prevent re-commits:**
   ```
   .env
   .env.local
   local.settings.json
   *.key
   *.pem
   ```

4. **Commit the cleanup with a clear message** (do NOT include the deleted values in the commit message):
   ```bash
   git add -A
   git commit -m "Remove literal secrets; migrate to Key Vault references

   - Removed hardcoded DB password, API keys, and connection strings
   - Replaced with @Microsoft.KeyVault() references
   - All secrets rotated in Key Vault and source systems
   
   Fixes SEC-002"
   ```

5. **Force-push or rebase to remove from history** (optional but recommended for production):
   ```bash
   git filter-branch --force --index-filter \
     'git rm --cached --ignore-unmatch <filename>' \
     --prune-empty --tag-name-filter cat -- --all
   ```

## Verification
1. Run threadlight: `python3 scripts/production_ready.py --target-rg <RG> --target-sub <SUB>`. SEC-002 should flip from `must-fix` to `pass` (no literal secrets found).
2. **Verify rotation completed:** Confirm all rotated secrets are now live in Key Vault with new values. Check:
   ```bash
   az keyvault secret show -n <secret-name> --vault-name <kvname> --query properties.updated -o tsv
   ```
   Timestamp should reflect the rotation, not the commit.

3. If threadlight still fails after cleanup, a leftover secret may remain in history or uncommitted changes. Re-run the regex sweep locally:
   ```bash
   grep -r "password\|token\|key" . --exclude-dir=.git
   ```
