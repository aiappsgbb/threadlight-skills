## Target file
`mcp-lock.json` at the repository root (a committed lockfile that records the exact MCP surface — server coordinates plus tool description and input-schema hashes — that was reviewed and approved).

## Edit type
`auto-apply` — the producer authors the lockfile for you; a remediation agent may generate and commit it. The assessor itself only reads the repo and reports whether the lock is present and matches the current surface.

## Edit recipe
`SUP-012` fires in two situations. Handle whichever applies:

1. **No lock committed** — generate one from the current, reviewed MCP surface and commit it:
   ```bash
   python3 scripts/mcp_sbom.py --root . --out mcp-sbom.json \
     --lock mcp-lock.json --update-lock
   git add mcp-lock.json
   ```
   Review the generated `mcp-lock.json` before committing — it is your approved baseline. Only commit it once the servers and tools it records are the ones you intend to ship.

2. **Lock present but drifted** — the live surface no longer matches the lock (a server version/digest changed, a package identity changed, or a tool's description or input schema changed). The scan detail names the specific drift. For each drifted item:
   - If the change is **intended** (you deliberately upgraded a server or accepted a new tool), re-author the lock and record why in the PR description:
     ```bash
     python3 scripts/mcp_sbom.py --root . --out mcp-sbom.json \
       --lock mcp-lock.json --update-lock
     ```
   - If the change is **unexpected**, treat it as a supply-chain event: stop, investigate the source of the changed server/tool, and only re-lock once you have confirmed the new surface is trustworthy.

Wire the same producer into CI so drift is caught on every pull request rather than at release time — the `--check` exit code gates the build (see the cicd `--mcp-gate` knob).
