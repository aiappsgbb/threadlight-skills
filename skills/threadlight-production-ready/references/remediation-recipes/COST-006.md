---
kind: manual
summary: Review cost recommendations in docs/cost-projection.md and apply the accepted SKU changes in infra/main.bicep, then re-run threadlight-consumption-iq to refresh the manifest.
---

## Target file
`infra/main.bicep` (or the relevant module under `infra/`) for SKU changes.
`specs/cost-manifest.json` is regenerated by `threadlight-consumption-iq` after changes are deployed.

## Edit type
`replace` — swap the resource SKU (e.g., PAYG tier → PTU tier, or downsize an ACA container resource) as recommended.

## What COST-006 checks

`COST-006` walks `specs/cost-manifest.json → recommendations[]` and reports
each entry where the deployed SKU still matches `current_sku` (i.e., the
recommendation has not yet been applied):

| Savings threshold | Status emitted |
|---|---|
| `monthly_savings_usd > $100` | `must-fix` |
| `monthly_savings_usd > $25` | `should-fix` |
| `monthly_savings_usd ≤ $25` | `pass` (rolled into report; advisory only) |

When `specs/cost-manifest.json` does not exist or has no `recommendations[]`
key, `COST-006` emits `not-verified` with the message:
> "Run threadlight-consumption-iq to populate cost-manifest.json before scoring COST-006."

## Edit recipe

1. **Open `docs/cost-projection.md`** and read the recommendations section.
   Each recommendation includes:
   - The resource's logical Bicep name
   - `current_sku` vs `recommended_sku`
   - `monthly_savings_usd` and `caveats` (e.g., PTU quota, commitment period)

2. **Decide which recommendations to accept.** Not all recommendations must
   be applied. Reasons to defer:
   - PTU quota not available in the required region
   - Monthly commitment incompatible with project timeline
   - Customer requires approval for capacity reservations

3. **For each accepted recommendation**, edit the relevant Bicep module
   (typically `infra/main.bicep` or `infra/modules/<resource>.bicep`):
   - Find the resource block by `logical_name`
   - Replace the `sku.name`, `sku.tier`, or `capacity` property
   - Example: change `sku: { name: 'gpt-4o', tier: 'PAYG' }` to
     `sku: { name: 'gpt-4o', tier: 'PTU', capacity: 25 }`

4. **Re-run the deployment** so the change lands in Azure:
   ```
   azd up
   ```

5. **Re-run `threadlight-consumption-iq`** to regenerate the manifest
   with the updated deployed SKUs. The recommendation should disappear
   (or reduce in savings) when the new SKU is the current deployment:
   ```
   python3 skills/threadlight-consumption-iq/scripts/consumption_iq.py run --all
   ```

6. **Re-run `threadlight-production-ready`** to verify `COST-006` now passes:
   ```
   python3 scripts/production_ready.py --target-rg <RG> --target-sub <SUB> --static
   ```

## Why this is `kind: manual` and not auto-applied

Cost recommendations involve capacity commitments (PTU units, reserved
instances) and SKU changes with trade-offs (latency, quota, billing model)
that require explicit human judgment before deployment. Applying a PTU
reservation without the customer's sign-off could create unexpected
billing commitments. The remediation is surfaced as an advisory checklist,
not an automatic Bicep patch.

## Verification

After completing the steps above, re-run the assessor and confirm:
- `COST-006` status flips to `pass` (or `should-fix` if lower-savings
  recommendations remain)
- `specs/cost-manifest.json → totals.monthly_savings_potential_usd` reflects
  the accepted savings

## Stale-plan check (the agent MUST do this before applying)

Recompute `sha256(canonical_json(<current production-readiness-manifest.json>))` and compare against `apply_plan["manifest_sha256"]`. If they differ, the plan is stale — refuse to apply and ask the operator to re-run `--onboard` to get a fresh apply-plan. See SKILL.md "Stale-plan detection" for the full contract.
