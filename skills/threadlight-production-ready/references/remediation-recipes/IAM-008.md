## Target file
`infra/rbac.bicep` (or the compiled `*.json` / any file declaring
`Microsoft.Authorization/roleAssignments` or Microsoft Graph `appRoles` /
`requiredResourceAccess` for the agent's identity).

## Edit type
`replace`

## Edit recipe
A workload identity holding **Owner**, **Contributor**, or **User Access
Administrator** — or a wildcard Graph permission like `Directory.ReadWrite.All` —
can do far more than triage a claim or answer a query. Grant only the specific
data-plane roles the agent needs, scoped to the specific resources it touches.

1. **Replace privileged built-in roles** with scoped data-plane roles:

   ```bicep
   // BEFORE (BAD) — Owner on the resource group:
   roleDefinitionId: subscriptionResourceId(
     'Microsoft.Authorization/roleDefinitions',
     '8e3af657-a8ff-443c-a75c-2fe8c4bcb635')  // Owner

   // AFTER (GOOD) — only what the agent uses, scoped to the resource:
   roleDefinitionId: subscriptionResourceId(
     'Microsoft.Authorization/roleDefinitions',
     '4633458b-17de-408a-b874-0445c86b69e6')  // Key Vault Secrets User
   scope: keyVault.id                          // not the whole RG/sub
   ```

   Common least-privilege swaps: **Key Vault Secrets User** (not Contributor),
   **Storage Blob Data Reader/Contributor** (not Owner), **Cognitive Services
   OpenAI User** for model calls, **AcrPull** (not AcrPush) at runtime.

2. **Remove wildcard Graph permissions.** Replace `Directory.ReadWrite.All` /
   `Application.ReadWrite.All` with the narrowest scope the agent actually calls
   (e.g. `User.Read.All`), and prefer delegated over application permissions where
   a signed-in user is present.

3. **Scope every assignment** to the individual resource id, not the resource
   group or subscription, so the grant cannot spread.

4. If you manage roles through **azure-rbac** or **foundry-agt**, encode the
   scoped roles there — the platform provisions least-privilege by construction.
